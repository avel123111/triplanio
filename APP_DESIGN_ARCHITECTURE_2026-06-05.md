# Triplanio — архитектура дизайна всего приложения

**Дата:** 2026-06-05 · **Рамки:** дизайн-система Triplanio (`Triplanio Design System/`) — Inter/Inter Tight, Atlantic (синий `#2167e2` + холодный slate + терракота как редкий акцент), monoline-иконки, без эмодзи, cool-slate тени, тон Linear/Arc, сдержанность вместо декора. **Цель:** одна целостная структура, в которой удобно пользоваться ВСЕМ функционалом, и которая красива за счёт точности, а не украшательства.

Это не «новая эстетика» и не набор экранов. Это: (1) полная карта функционала, (2) единая навигация и где живёт каждая возможность (с переразмещениями), (3) единая система оболочки/сетки/панелей, (4) один компонентный словарь из твоей системы. Визуал — следствие, и он будет один на всё приложение.

---

## 1. Полная карта функционала (чтобы ничего не потерять)

**Аккаунт-уровень (вне трипа):**
- Коллекция трипов (активные/прошлые, сетка/список, поиск, создание).
- Инбокс — приглашения, голосования, обновления.
- Pro / тарифы (подписка мес/год + разовый Pro для трипа).
- Аккаунт (профиль, язык, тема, биллинг/подписка).
- Лендинг (маркетинг), Login/signup/reset.

**Создание трипа:** ручной планнер (Откуда → Скелет городов → Возврат → Финальный драфт) и AI-планнер (описал словами → черновик).

**Трип-уровень — линзы (взгляды на один трип):**
- Хронология (лента событий по дням).
- Карта (маршрут, точки городов).
- Календарь (месяц/неделя, многодневные город-бары).
- Бюджет (категории/города, ручные траты, FX-курсы, сплит на участников).
- Документы (общие + личные).
- Чат (групповой + @Triplanio AI-ассистент).
- Выбор отелей (голосование — «скоро»).

**Трип-уровень — управление:**
- Редактор структуры (города, переезды, ночи, старт/финиш — песочница).
- Участники (роли владелец/админ/зритель, приглашения).
- Настройки (название, валюта отображения, варнинги, опц-фичи, аппруверы голосования, опасная зона).
- Поделиться (ссылка / публичный трип).

**Сущности внутри трипа:** отели (заезд/выезд, дедлайн отмены, статус оплаты, ваучеры), переезды (рейс/поезд, пересадки-сегменты, ночной флаг), активности, аренда авто (получение/возврат), услуги (eSIM, авто, страховка — партнёрские платформы), города (даты/ночи).

**AI:** @Triplanio в чате (правит трип с согласия владельца), AI-парсинг ваучеров/броней (в точке добавления), AI-планнер.

**Интеграции:** Telegram-бот (напоминания, many-to-many чат↔трипы), WhatsApp (скоро), Stripe (Pro).

**Сквозное:** варнинги/валидация (нет переезда, нет брони, даты вне окна, дедлайны), мультивалюта, i18n EN/RU/ES, роли/доступы, копирование трипа, публичный просмотр, Free/Pro-гейтинг.

## 2. Корень проблемы — структура, а не пиксели

Сегодня **почти каждый экран — своя оболочка и свой layout** (это я подтвердил живым обходом):
- В трипе — лейбл-рейл линз + хлебная крошка.
- Редактор структуры — ДРУГАЯ оболочка: collapsed icon-рейл + своя топ-панель действий + единственная в приложении таблица.
- Аккаунт-уровень (Trips/Pro/Аккаунт/Инбокс) — рейла нет, центрованный контент, другой хедер.
- Правый рейл то 320px (Хронология), то 280px (Чат), то маршрут-панель (Карта), то исчезает (Бюджет/Календарь).
- Одна сущность открывается то модалкой (событие в ленте), то левой in-place панелью (город в редакторе), то инлайн-карточкой.

Отсюда «разрозненно и дёшево», даже когда отдельный экран чистый: **нет единого скелета.** Красота в системе Triplanio = единый каркас + сдержанные компоненты, а не декор. Значит редизайн — это в первую очередь **дать приложению один скелет**, и уже в нём — твои компоненты.

## 3. Единая архитектура

### 3.1 Одна оболочка (App Shell) для всего

Один постоянный каркас, который не меняется между разделами:

```
┌───────────────────────────────────────────────────────────────┐
│ TOPBAR (72px, фикс): [◧ Triplanio] [ / Трип · даты ]  ⌕  ◔inbox  ☾  (avatar) │
├──────────┬────────────────────────────────────────────────────┤
│ NAV-RAIL │  CONTENT (container 1180, gutter 24)                │
│ (один    │  ┌──────────────────────────────┬─────────────────┐ │
│  рейл,   │  │  MAIN (линза/раздел)          │  CONTEXT-RAIL    │ │
│  220px,  │  │                               │  (постоянный,    │ │
│  сворач. │  │                               │   336px)         │ │
│  в 64px) │  └──────────────────────────────┴─────────────────┘ │
└──────────┴────────────────────────────────────────────────────┘
```

- **Topbar** — всегда один: бренд-марка `assets/logo-mark.svg` (28px), крошка трипа (когда внутри трипа), глобальные действия (поиск, инбокс, тема, аватар). Скролл → frosted (`rgba(255,255,255,.78)` + blur, как в системе).
- **Nav-rail** — ОДИН рейл с двумя контекстами (не две разные оболочки):
  - *Аккаунт-контекст* (вне трипа): Трипы · Инбокс · Pro · Аккаунт.
  - *Трип-контекст* (внутри трипа): группа **Линзы** (Хронология/Карта/Календарь/Бюджет/Документы/Чат) + группа **Управление** (Структура/Участники/Настройки/Поделиться). Active-state = `--brand-soft` + `--brand` (как в системе). Сворачивается в icon-rail на узких — но это ТА ЖЕ оболочка, иконки не прыгают.
- **Content** — всегда `container 1180 / gutter 24`. Главная колонка + опциональный **постоянный** правый рейл фикс-ширины.

→ Это убивает 3 разные рамки. Редактор структуры перестаёт быть отдельным приложением (см. 3.2).

### 3.2 Где живёт каждая возможность (переразмещения)

| Возможность | Сейчас | Решение в единой архитектуре |
|---|---|---|
| **Редактор структуры** | Отдельная full-screen оболочка + таблица + своя топ-панель | Становится **линзой «Структура»** в том же shell. Главная колонка = маршрут карточками (НЕ таблица), правый CONTEXT-RAIL на этой линзе = живая карта (она тут контекстна). Действия Сохранить/Сброс — в topbar справа, не отдельной панелью. |
| **Правый рейл** | Разный/исчезает по линзам | ОДИН **CONTEXT-RAIL «досье трипа»** постоянной ширины 336px на ВСЕХ линзах: обложка, даты, города, мини-бюджет, кто едет, счётчик варнингов. Это якорь, который держит ощущение «это один трип» при переключении линз. Линзо-специфика (детали бюджета, полная карта) — в главной колонке, не в рейле. |
| **Карта** | И линза, и встроена в редактор по-разному | Линза «Карта» = полноэкранная карта + маршрут-инфо в главной колонке. В линзе «Структура» карта = context-rail. Один компонент карты, два места. |
| **Услуги (eSIM/авто/страховка)** | Виджет только в правом рейле Хронологии | Переезжают в точку смысла: кнопка-добавление прямо в дне/городе ленты (контекстно) + единый раздел «Услуги» как под-вид Бюджета/Документов (где они логически = расходы+брони). Убрать из глобального рейла. |
| **AI-консьерж** | Чат-линза + плавающий dock + AI-парсинг ваучеров отдельно | ОДНА AI-сущность @Triplanio: (а) парсинг ваучера — инлайн в потоке добавления брони (в панели сущности, см. 3.3); (б) разговор — линза Чат; (в) быстрый доступ — один постоянный вход в topbar/рейле, не случайный dock. Везде один лиловый AI-сюрфейс. |
| **Варнинги/валидация** | Разрозненные баннеры по экранам | Единая поверхность: счётчик в CONTEXT-RAIL (досье) → клик открывает **панель «Что требует внимания»** (список варнингов с переходом к сущности); инлайн-маркеры в ленте остаются как указатели. |
| **Pro-гейтинг** | Разные локи (модалка/карточка/диалог) | Один компонент **ProLock** (панель/состояние) для всех закрытых фич — единый текст/визуал. |
| **Инбокс / Уведомления** | Аккаунт-уровень, своя центровка | Тот же shell (аккаунт-контекст), список = `tl-card`-подобные строки. |
| **Создание трипа** | Свой flow-shell | Остаётся отдельным фокус-режимом (это правильно — флоу), но в том же визуальном языке: topbar-прогресс, главная колонка-форма, context-rail = живая карта/превью. Manual и AI — один каркас, разный контент шага. |

### 3.3 Одна система панелей детали/редактирования

Любая сущность (событие, город, отель, переезд, активность, трата, документ, участник) открывается в **ОДНОЙ правой панели** (slide-in над scrim), а не в трёх разных контейнерах:

- **Просмотр → Редактирование в одной панели** (toggle), без отдельных модалок.
- Структура панели одинаковая: шапка (тип + название), стат-строка ключевых полей, секции (детали/финансы/контакты/документы), действия внизу.
- AI-парсинг ваучера живёт ВНУТРИ панели создания брони (вставь текст/файл → поля заполнились), а не отдельным экраном.
- Один паттерн = предсказуемость + конец «разнотипности панелей».

### 3.4 Одна content-сетка

`container 1180`, `gutter 24`, главная колонка + **CONTEXT-RAIL 336px постоянной ширины** (опционален: на маркетинг/аккаунт-экранах его нет, контент центрируется в том же контейнере). Никаких «то 320, то 280, то нет».

## 4. Один компонентный словарь (из твоей системы — везде)

Всё приложение собирается из ОДНОГО набора (твои `preview/*`), а не пересобирается на каждом экране:

- **`tl-card`** — строка события/списка: плоская белая, 1px `--line`, радиус 10, monoline-иконка 1.6, `<strong>LHR → LIS</strong> · British Airways 503`, тег категории, время tabular. Это базовый ряд ВЕЗДЕ (лента, инбокс, документы, участники, траты).
- **`tag`** по категории: Flight=`--brand`, Hotel=`--success`, Activity=`--warm`. Один маппинг во всём приложении.
- **`budget-bar`** — крупное число Inter Tight + сегмент-метр + легенда-точки + «€180 under plan»-пилюля.
- **tabs** (белая пилюля-актив), **buttons** (primary/ghost из системы), **chips/eyebrows**, **icon-tile 42×42 `--brand-soft` r11**, **empty-states** (иконка+строка+CTA), **CONTEXT-RAIL «досье»**, **ProLock**, **AI-сюрфейс**.
- **monoline-иконки** `reference/icons.jsx` (`<Icon name="bed" size={20}/>`). **Эмодзи/юникод запрещены.**

Правило: новый экран = композиция этих компонентов в общей сетке. Если нужен новый компонент — он добавляется в словарь и переиспользуется, а не живёт на одном экране.

## 5. Визуальный язык (красота через сдержанность)

Строго твоя система, без моих выдумок:
- **Шрифт:** Inter Tight (дисплей) / Inter (тело). Никакого серива по умолчанию.
- **Цвет:** синий `#2167e2` делает 80% работы; холодный slate; терракота — только активности/фото-моменты; зелёный success — статусы. Фон белый / `--wash`. Никаких градиентных «обложек» и шумов.
- **Фото:** тёплые golden-hour, затемнённые градиентом-оверлеем (как в баннере системы), никогда не «голое большое фото».
- **Тени:** `--shadow-soft/card/pop` (cool slate). Радиусы 18/12/10/999. Граница 1px `--line`.
- **Движение:** reveal-on-scroll (opacity+`translateY 12px`, 600ms), hover карточки `translateY(-2px)`+`--shadow-card`+бордер→`#dbe1ec`, hover кнопки `translateY(-1px)`+цветная тень, табы→белая пилюля, AI-точки. Без пружин/баунсов.
- **Тон:** Linear/Arc, sentence case, em-dash несёт голос, `·`-разделители, `→` для рейсов/дат, валюта впереди tabular, без восклицаний/эмодзи.

## 6. Экран-архетипы в единой системе (как всё ложится)

Все экраны — это 5 архетипов одного shell, а не 15 уникальных layout'ов:

1. **Коллекция** (Трипы, Инбокс) — аккаунт-контекст, главная колонка с `tl-card`/карточками трипов, без context-rail.
2. **Трип-shell + линза** (Хронология/Карта/Календарь/Бюджет/Документы/Чат/Структура) — nav-rail трипа + главная колонка (контент линзы) + постоянный CONTEXT-RAIL «досье». Линза меняет только главную колонку.
3. **Панель сущности** — правый slide-in над любым из вышеперечисленных (просмотр/редактирование). Один на всё.
4. **Фокус-флоу** (создание трипа manual/AI, чекаут Pro) — topbar-прогресс + форма + превью-rail. Отдельный режим, тот же язык.
5. **Маркетинг/Аккаунт** (Лендинг, Pro, Аккаунт, Login) — центрированный контент в том же контейнере, секции системы (eyebrow/h2/lede/feature-cards).

## 7. Конкретные IA-решения (что переосмыслить)

1. **Редактор структуры → линза «Структура»** в общем shell (карта в context-rail, действия в topbar). Минус одна чужая оболочка и единственная таблица.
2. **CONTEXT-RAIL «досье трипа» постоянный** на всех линзах — главный носитель «это один трип».
3. **Варнинги → единая поверхность** (счётчик в досье + панель «Что требует внимания» + инлайн-маркеры).
4. **AI → одна сущность** (инлайн-парсинг в панели брони + чат-линза + один вход), единый AI-сюрфейс.
5. **Услуги → в контекст** (день/город ленты + раздел в Бюджете), не в глобальный рейл.
6. **Панели сущностей → один правый slide-in** (конец модалка-vs-инплейс-vs-инлайн).
7. **Pro-локи → один ProLock**.
8. **Один компонентный словарь** (tl-card/tag/budget-bar/…); запрет «локальных» переизобретений.
