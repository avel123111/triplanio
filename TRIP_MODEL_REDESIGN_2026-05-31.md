## 1. Цель и главный принцип

Дать гибкость редактирования уже заполненного трипа, не теряя структуру и варнинги.

Принцип: **связи событий с городом не хранятся жёстким FK на строку маршрута, а ВЫЧИСЛЯЮТСЯ** из самодостаточных данных события. Каждое событие (отель, активность, трансфер) несёт: свой **город как стабильное значение** (`city_ref`), своё **время + таймзону**, своё **место**. Принадлежность к городу и покрытие переходов — функции от (`city_ref` + даты), а не FK.

Из этого:
- структура = явный список городов, упорядоченный по датам, + переезды со своими концами;
- гибкость = все даты/времена независимы, пересечения разрешены, **ничто не блокируется, не мутируется молча и не удаляется каскадом**;
- логика = варнинги, посчитанные поверх (никогда не блок).

## 2. Модель данных (конкретно: keep / +add / поведение)

### city_visits (города/остановки)
- keep: `city_name, country, country_code, latitude, longitude, timezone, start_datetime, end_datetime, kind, notes`
- **+ `city_ref` (text)** — стабильный ключ города (Google `place_id`). Заменяет ненадёжный `external_city_id` (Nominatim `place_id`, нестабилен/часто null).
- порядок городов = `ORDER BY start_datetime` (см. §9 про якоря/без-дат). `position` НЕ вводим.
- даты = окно пребывания, своё, свободно пересекается с соседями.

### hotel_stays
- keep: `name, address, latitude, longitude, check_in_datetime, check_out_datetime, booking_reference, payment_status, free_cancellation, ...`
- **+ `city_ref` (text)** — город отеля (значение-снимок, проставляется при создании; обычно наследуется от города, под которым добавляют).
- **+ `city_name, country_code` (денормализация для рендера без вызовов провайдера).**
- **+ `timezone` (text)** — из координат отеля (для корректного ремайндера чек-ина).
- `city_visit_id` — **перестаёт быть FK-владельцем**; см. §7 (убрать `ON DELETE CASCADE`). Группировка под героем = по `city_ref` + перекрытие дат.
- `is_booked` — производное: есть `booking_reference`/подтверждён → **якорь** (молча не двигается).

### activities
- keep: `title, start_datetime, end_datetime, location_address, location_latitude, location_longitude, price, ...`
- **+ `city_ref`, `city_name`, `country_code`, `timezone`** (аналогично отелю).
- `city_visit_id` — как у отеля (убрать CASCADE).

### transfers
- keep: `from_address, to_address, from_latitude/longitude, to_latitude/longitude, start_datetime, end_datetime, transport_type, carrier, flight_number, booking_reference, ...`
- **+ `from_city_ref`, `to_city_ref` (text)** — города концов (значения-снимки).
- **+ `departure_timezone`, `arrival_timezone`** — из координат концов (кросс-tz перелёты, корректные длительность и ремайндер).
- **+ `legs` (jsonb)** — для составного переезда: упорядоченные сегменты, у каждого свои `from/to_place` + времена; пересадка = граница ноги; «провести день» → промоут пересадки в `waypoint`-город.
- `from/to_city_visit_id` — депрекейт (сейчас `SET NULL` при удалении города); матч переезда к переходу — по `city_ref`-паре + времени.

### Время (фундамент)
- Сейчас: наивный wall-clock (`time.js`/`naive-time.js`), `city.timezone` в рендере игнорируется, в валидации используется. Из-за этого **ремайндеры считаются по UTC-цифрам и врут на оффсет города** (см. §7).
- Целевое: каждое датированное поле хранит **локальное время + свою IANA-`tz`** (паттерн уже есть у аренды авто: `pickup_at_local` + `pickup_timezone`). Истинный инстант вычисляется на лету (для ремайндеров/сортировки). UX «ввёл 15:00 — видишь 15:00» сохраняется (дисплей = инстант в `tz` события).

### Провайдер геоданных
- Стандартизируемся на **Google** (`placesAutocomplete` edge fn уже подключён: autocomplete + details + timezone; `place_id` стабилен; города и адреса — одно пространство id).
- `city_ref` = Google `place_id`; `tz` = Google Timezone API по координатам.
- Nominatim из браузера (`geo.js searchCities`) **выпиливаем** (нестабильный `place_id`, usage-policy на масштабе).

## 3. Матчинг и покрытие (вычисляемые правила)

Все правила — **советующие варнинги**, никогда не блок.

### Отель ↔ город
Матч: `hotel.city_ref == visit.city_ref` И даты пересекаются (`overlap(hotel[ci,co], visit[start,end])`). Дальше арифметика дат:
- полное совпадение → рендер под героем, без замечаний;
- заезд позже начала визита → рендер + «гэп до заезда» (start..checkin);
- выезд позже конца визита → рендер + «лишняя бронь» (end..checkout);
- **нет пересечения** ни с одним визитом этого города → не рендерим под героем; в городе варнинг «нет жилья на эти даты»; бронь уходит в блок **«неиспользованные брони»** (ничего не теряется).
- город важен, чтобы мадридский отель не приклеился к Риму по одному перекрытию дат (матч = город И даты).

### Переезд ↔ переход
Для каждой пары соседних по датам визитов A→B с разными `city_ref`: матч если есть transfer с `from_city_ref==A.city_ref` И `to_city_ref==B.city_ref` И время попадает в **окно стыка** (см. §6 — правило окна). Иначе:
- стык без переезда → варнинг «нет переезда A→B»;
- переезд, не легший ни на один стык → блок **«неиспользованные переезды»**.

### Активность ↔ город
Матч по `city_ref` + попадание времени в окно визита; вне окна → варнинг.

## 4. Сценарии построения трипа

- **Ручной мастер (`ManualPlanner`)**: города с датами; каждый узел захватывает `city_ref` + `tz`; переезды создаются с `from/to_city_ref`. Каскад дат `recomputeDates` — оставить только как удобство (не как жёсткую цепь).
- **ИИ (`planTripWithAi`)**: LLM выдаёт **имена**, не id. Добавляем **resolve-проход**: каждое имя → Google geocode → `{city_ref, координаты, tz, каноническое имя}` → сохраняем в `city_visit`/события. Чинит сегодняшнюю дыру (у ИИ-трипов `external_city_id` пуст, координат часто нет). Дизамбигуация: ИИ отдаёт «город, страна», резолвер берёт топ по стране/важности.
- **Пустой трип**: города без дат живут вне таймлайна (трей «без дат»), на хронологию не лезут.

## 5. Сценарии редактирования (корнер-кейсы и поведение)

Город A=Madrid, B=Rome. Везде: edit проходит всегда, ничего не теряется/не блокируется.

1. **Сдвиг выезда A +1 / въезда B −1** → меняется только расписание. Переезд остаётся `Madrid→Rome`. Максимум варнинг «вылет позже выезда из A». Ничего не рушится.
2. **Наслоение городов** (A и B пересеклись) → разрешено, варнинг «города пересекаются».
3. **Реордер** (B раньше A) → меняешь даты (порядок = даты). Брони по умолчанию едут с городом (превью+undo); **подтверждённая бронь не двигается** → варнинг «бронь вне пребывания».
4. **Вставка города между A и B** → новый визит; существующий переезд `Madrid→Rome` концы НЕ меняет → варнинг «маршрут не сходится: между Madrid и Rome теперь Florence — это пересадка или рейс не подходит?». Никакого молчаливого переподключения.
5. **Удаление среднего города** → его отели/активности (матч по `city_ref`) становятся **сиротами** → трей «неиспользованные брони», город исчез. Переезды с его `city_ref` → сироты + варнинг. **Критично: сегодня это CASCADE-удаление брони (см. §7).**
6. **Отель: гэп / лишний день / нет покрытия** → по §3 (рендер+гэп / рендер+лишнее / трей сирот).
7. **Отель на стыке двух городов** (Madrid 13–16, hotel Madrid 15–18) → матч по `city_ref=Madrid`, рендер под Madrid + «лишняя бронь 16–18» (в Rome НЕ показываем — другой `city_ref`).
8. **Переезд: нет / не та дата / не то направление** → §3 (нет переезда / сирота).
9. **Составной переезд (2 перелёта)** → `from/to_city_ref` = концы, `legs` внутри; пересадка-точка опционально промоутится в `waypoint`.
10. **Один город дважды** (Madrid рано и поздно) → `city_ref` одинаков; какой визит покрывает отель — решается перекрытием дат.
11. **Кросс-tz переезд** (Madrid→Tokyo) → `departure_timezone`/`arrival_timezone`, истинные инстанты → верная длительность и ремайндер.
12. **Бронь vs план** → `is_booked` бронь при сдвиге города стоит + варнинг; план-элемент едет.
13. **Город без дат** → трей «без дат», не на таймлайне.
14. **Смена города у визита** (перевыбрали город) → `city_ref` меняется → его отели (матч по старому `city_ref`) становятся сиротами + варнинг. Предложить «перепривязать брони».
15. **DST / переезд через полночь** → истинные инстанты считаются по `tz`, длительность корректна.

## 6. Валидации (как работают; все советующие)

- **Покрытие отеля**: для каждого отеля найти визиты с тем же `city_ref`, пересекающиеся по датам; нет → сирота; есть → посчитать гэп (`visit.start < checkin`) и лишнее (`checkout > visit.end`).
- **Покрытие переезда**: для соседних по датам визитов A→B (разный `city_ref`) искать transfer `from=A,to=B` в окне стыка. **Окно стыка**: города впритык (`A.end == B.start`) → точка этого дня; есть разрыв (`A.end < B.start`) → интервал `[A.end, B.start]` с допуском (напр. ±1 день). Нет → «нет переезда»; transfer без стыка → сирота.
- **Активность**: `city_ref` + окно визита.
- **Пересечение визитов**: даты двух визитов пересекаются → варнинг (не блок).
- **Якорь брони**: `is_booked` элемент вне окна своего города → варнинг + подсказка «перебронировать», **без авто-движения**.
- **Темпоральная связность**: прилёт раньше вылета; время переезда вне стыка; ремайндер не считается (нет `tz`) → скип + флаг (как сейчас у авто).

## 7. Риски с учётом текущей реализации

1. **CASCADE-удаление = потеря данных уже сейчас.** `hotel_stays.city_visit_id` и `activities.city_visit_id` — `ON DELETE CASCADE`; `transfers.from/to_city_visit_id` — `SET NULL`. Удаление города молча сносит брони. **Обязательно** убрать CASCADE (на `SET NULL`/`RESTRICT` + наш диалог) — это и баг, и предпосылка модели сирот.
2. **Миграция времени.** Существующие даты лежат как наивный-as-UTC. Бэкофилл истинных инстантов требует `tz` на запись — резолвим из координат (у визитов/отелей/активностей/трансферов координаты есть). Без координат → фолбэк tz города / home, иначе ремайндер скип. Риск: неверные инстанты при кривом резолве.
3. **`external_city_id` нестабилен/часто null.** Бэкофилл `city_ref`: переразрешить через Google или матчить переходный период по `city_name+country_code`. Риск смешанных данных.
4. **Ремайндер-SQL переписать.** `get_pending_reminders` и `get_*_tomorrow` (миграция `add_daily_reminder_functions`) считают по UTC `::date`/наивно. Нужно на истинные инстанты + `tz` события. Сохранить n8n-расписание и дедуп `telegram_reminder_logs`.
5. **Развязка ломает всех FK-читателей.** `hotelsByVisit`/`inboundByVisit` в `TripView`, `HotelTimeline`, `getTripDetails`, `copyTrip` (ремапит `city_visit_id`), `exportTripPdf` (base44), `segments.js` (календарь). Везде заменить join-по-FK на матч `city_ref`+даты.
6. **Два провайдера → один.** Перевод города с Nominatim на Google: переписать `geo.js`, точки выбора города (`CitySearch`, `ManualPlanner`, `CityVisitDialog`).
7. **Два Supabase-проекта + dev/main.** Миграции и edge-функции катить на оба; `verify_jwt` для webhook-функций не сбрасывать.
8. **Бюджет.** `syncTripExpense`/budget autosync завязаны на id сущностей, не на город → вероятно не страдают; проверить.
