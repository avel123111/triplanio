## 0. Контекст и проблема

Сегодня трип нельзя гибко править после наполнения: любое изменение дат/городов мгновенно валидируется и каскадит против всего остального, из-за чего «двигаешь одно — рушится другое». Конкретные источники жёсткости в текущем коде:

- `CityVisitDialog.jsx:172-175` — `canSave` блокирует сохранение города при ЛЮБОМ пересечении дат с соседом (`overlapWith`, стр. 93-105). Нельзя сдвинуть выезд из A, не подвинув B.
- Порядок городов выводится из дат (`sortVisits`, `validation.js:174`), `position` нет.
- Трансфер жёстко привязан к двум городам (`from_city_visit_id`/`to_city_visit_id`, `EventEditDialog.jsx:945-946`); составной = N строк с одинаковыми концами + текст в адресах.
- Удаление города (`TripView.jsx:1499-1507`, `confirmDeleteCity`) немедленно и безвозвратно сносит все отели/активности/переезды явными DELETE. NB по схеме (проверено dev): FK `hotel_stays.city_visit_id` и `activities.city_visit_id` = `ON DELETE CASCADE`, а `transfers.from/to_city_visit_id` = `ON DELETE SET NULL` (НЕ cascade) — прямое удаление города отели/активности снесёт, а трансферам обнулит концы → останется сирота-трансфер. Поэтому штатный путь — явные DELETE (см. §6).
- Варнинг «нет переезда между соседями» живёт в рендере (`hasTransferBetween`, `TripView.jsx:787`), а не в данных — в песочнице (без таймлайна) его нет.

## 1. Решение (суть)

Отдельный **экран комплексного редактирования** («песочница»). Пока пользователь в нём — НИКАКИЕ изменения в БД не вносятся, всё состояние держится в памяти/`sessionStorage`, вся логика вычисляется на лету. Экран показывает города как блоки с датами и переезды между ними (без таймлайна). На экране доступны: добавление/удаление/переупорядочивание городов, правка дат, правка/добавление/удаление событий (отели, активности, переезды).

Во время редактирования считаются **валидации** (раздел 5). Пока есть хотя бы одна ошибка/варнинг — кнопка «Сохранить» заблокирована. Когда чисто — «Сохранить» делает **batch-запись** (upsert по id + явные удаления, раздел 6) в БД.

Каркас данных НЕ меняется: `hotel→city_visit_id`, `activity→city_visit_id`, `transfer→from/to_city_visit_id` остаются. Без `city_ref`, без смены гео-провайдера. Это слой сверху, а не передел модели. **Единственное добавление в схему контента — `city_visits.position`** (§4a): НЕ свободный пользовательский порядок (как в отменённом переделе), а авто-поддерживаемый тай-брейкер, ЖЁСТКО подчинённый датам — нужен, чтобы соседство (D5/E1/E3) и таймлайн детерминированно решали порядок узлов с равным/общим днём.

### Почему «нельзя сохранить с варнингами», а не «мягкие варнинги»
Таймлайн (`TripView`) физически не способен адекватно отрисовать произвольно несвязный трип (перелёт между городом 7 и 2, отель вне всех городов). Гейт «трип валиден на входе в таймлайн» — осознанный размен: жёсткость на сохранении в обмен на то, что таймлайн не нужно учить рисовать хаос. Песочница без таймлайна — место, где промежуточные несогласованные состояния допустимы, потому что они нигде не сохраняются и нечего рушить.

## 2. Доступ (гейтинг)

Правило (как сейчас в продукте — проверено по коду):
- **Роль:** редактировать может любой, **кроме `viewer`** (`myRole !== 'viewer'`, `TripView.jsx:1592`; кнопка «Редактировать» в `TripCoverStrip` гейтится так же, `:1198-1201`).
- **Активный vs прошлый трип:** «прошлый» = `isTripInPast(visits)` (`trip-dates.js:34-42` — общий `end` строго раньше сегодня; трип без дат считается активным).
  - **Активный трип** — редактировать можно (только условие роли выше).
  - **Прошлый трип** — редактировать можно, **только если трип Pro ИЛИ владелец Pro**, т.е. `tripIsPro` (`TripView.jsx:1621`: `!!trip.is_pro_trip || ownerProResolved`). Без Pro прошлый трип — read-only (сейчас это `isPastTrip`-блок в `TripFormDialog.jsx:68,121`).

Итоговое условие входа в Edit Mode:
```
canEditMode = myRole !== 'viewer' && ( !isTripInPast(visits) || tripIsPro )
```
NB: `tripIsPro` = Pro самого трипа ИЛИ активная подписка ВЛАДЕЛЬЦА (НЕ `isProActive(user)` смотрящего — это account-Pro для шапки, `:1600`, к гейту редактирования не относится).

Новых правил доступа не вводим — это ровно текущая модель. Pro в трипе сверх этого закрывает другое (линзы Календарь/Бюджет-разбивка/ИИ/Чат, `TripView.jsx:297,395,1968`), к гейту Edit Mode не относится.

## 3. Конкурентность (лок + таймаут)

Установлено по коду: трип-контент (`city_visits/hotel_stays/activities/transfers`) пишет **только UI**. Edge-функции/бот/n8n в эти таблицы НЕ пишут — только читают (исключения `copyTrip` — создаёт новый трип, `deleteMyAccount` — сносит аккаунт). Значит единственный конкурент — **второй человек в UI** в шаренном трипе. Optimistic-lock по версии/`updated_at` НЕ нужен (нет писателей в обход).

Механизм:
- Новые колонки на `trips`: `editing_by uuid NULL`, `editing_since timestamptz NULL`.
- Вход в Edit Mode: если `editing_by` пуст или `editing_since` старше **TTL = 30 мин** → захватить (`editing_by = me, editing_since = now()`). Иначе показать «Трип сейчас редактирует <имя>, попробуйте позже».
- Heartbeat: обновлять `editing_since = now()` каждые ~5 мин, пока экран открыт.
- Выход: явный (кнопки «Сохранить»/«Отмена») освобождает лок (`editing_by = null`). Закрытие вкладки — best-effort через `beforeunload` (не гарантирован) → подстраховка TTL: чужой лок старше 30 мин считается протухшим и перехватывается.
- **Известный технический долг (зафиксировать):** при протухшем TTL остаётся узкое окно «двое редактируют», где второй может затереть правки первого, т.к. финальной сверки версии нет (Pavel отказался от версии). Приемлемо на текущем этапе (prod-пользователь по сути один); при росте шеринга — вернуться к optimistic-lock по `updated_at`.

## 3a. Заморозка мутаций эвентов вне Edit Mode

Пока трип под редактированием (лок `editing_by` занят), любое **создание / изменение / удаление** эвентов (`hotel_stays`, `transfers`, `activities`) из ВСЕХ обычных точек входа должно быть заблокировано — иначе параллельная правка рассинхронит черновик песочницы с БД. **`trip_services` НЕ замораживаем** (решение Pavel 2026-06-01): они на уровне трипа, не привязаны к городу/маршруту, в песочнице не редактируются и в batch-save не входят — рассинхрона черновика не создают. **Сам вид таймлайна (просмотр) НЕ ограничиваем** — только мутации. Точки входа на блокировку (по коду):

- **TripView** (создание/выбор): `setHotelChoice`/`setHotelEdit`, `setTransferChoice`/`setTransferEdit`, `setActivityEdit`; `AddDayButton` (`onAddCity`/`onAddActivity`, `TripView.jsx:409`); warning-кнопки `MissingHotelWarning`/`MissingTransferWarning` (`onAddHotel`/`onAddTransfer`). (`setServiceChoice`/`setServiceEditCar`/`setServiceEditSimple` — НЕ морозим.)
- **TripView** (редактирование/удаление): `openEventView`→`EventEditDialog` (hotel/transfer/activity, внутр. «Удалить» `EventEditDialog.jsx:599`); `onDeleteCity`→`confirmDeleteCity` (`TripView.jsx:1499`). (`ServiceDialog` delete `:72` — НЕ морозим.)
- **Прямые delete-кнопки в компонентах:** `HotelTimeline.jsx:20`, `ActivityList.jsx:27`, `TransferStrip.jsx:30`. (`ServicesWidget` в правом рейле — НЕ морозим.)

Реализация: гейт по флагу «трип редактируется» (тот же `editing_by`/состояние) — кнопки add/edit/delete скрыты или disabled с подсказкой «Трип сейчас редактируется». Распространяется и на владельца, открывшего другую вкладку, и на других участников. (Существующий `isEditMode` в TripView — это инлайн-режим таймлайна, НЕ путать с Edit Mode-песочницей; на время песочницы инлайн-edit тоже недоступен.)

## 4. Модель экрана (поведение каркаса)

- **Города** — блоки с датами `[arrival … departure]`, перетаскиваемые (reorder), даты редактируемые. Порядок трипа = по `start_datetime`, а при равном/общем дне — по `position` (§4a). Reorder = пользователь меняет порядок блоков на экране: при разных датах это переустановка дат; при равных/общих датах (напр. однодневные `city2 12-12`, `city3 12-12`) — переустановка `position`. (Каркас можно частично переиспользовать из `ManualPlanner`: `CityRow`, `recomputeDates` — но это ОТДЕЛЬНЫЙ компонент, не пошаговый визард.)
- **Перемещение/каскад дат двигает ТОЛЬКО город (единое правило).** И «нудж даты» (`recomputeDates`-подобный каскад: сдвиг города пересчитывает даты последующих городов), и drag-reorder двигают только узлы-города. Привязанные **отели, активности и связанные трансферы НИКУДА не двигаются** — остаются на своих датах и падают в варнинг, если перестали сходиться с датами своего города (B/C/D); как только их даты поправили и они снова сходятся — варнинг гаснет. Авто-сдвига броней вместе с городом НЕТ. Допускается много варнингов от одного действия — упрощение каскада отложено.
- **События** (отели/активности/переезды) — редактируются прямо в песочнице; изменённый эвент уедет в БД обновлённым на сохранении.
- **Удаление города и любого эвента в песочнице НЕ выполняется сразу** — только пометка «удалить»; физический DELETE — на batch-save (раздел 6).

## 4a. Порядок узлов — `position` (новая колонка, авто-поддерживаемая)

Решение Pavel 2026-06-01: добавить `city_visits.position int`. Это **не** свободный пользовательский порядок (в отличие от отменённого передела модели) — это тай-брейкер, ЖЁСТКО подчинённый хронологии.

- **Инвариант:** `position` ВСЕГДА согласован с хронологическим порядком (`start_datetime`). Сценария «position разошёлся с датами» быть не должно ни при каких условиях.
- **Назначается автоматом, не пользователем.** Базовый порядок = sort по `start_datetime`; при РАВНОМ `start` (в т.ч. однодневные узлы и будущие waypoint, §11) тай-брейк берётся из порядка, который пользователь выстроил на экране, и фиксируется в `position`. Кейс-эталон: `city1 10–12, city2 12–12, city3 12–12, city4 12–14` — даты city2/city3 равны, их относительный порядок решает `position`.
- **Пересчёт:** при любой правке дат / reorder / добавлении / удалении узла — нормализовать `position` всего трипа одним проходом (0..N по `(start, текущий on-screen tie-order)`), чтобы инвариант держался.
- **Использование:** сортировка узлов везде = `(start_datetime, position)`. На неё переходят (а) валидаторы соседства D5/E1/E3; (б) **таймлайн `TripView` TimelineLens** — сейчас на чистом `sortVisits`, наследует ту же `(start, position)`-логику для корректной отрисовки порядка в пограничных случаях (закрывает остаток бага про общие дни из фикса timeline-order).
- **Зона влияния `position`:** миграция `city_visits +position` (prod+dev, бэкофилл существующих трипов из текущего `sortVisits`-порядка); `sortVisits` в `validation.js` принимает `position` в тай-брейк; `ManualPlanner`/`AiTripPlanner`/`copyTrip` проставляют `position` при insert (последовательно по порядку городов); `getTripDetails`/`getPublicTrip`/`tripPayload` отдают `position` (если select=`*` — протекает само, проверить); TimelineLens и `MapView`/`mapRoute` сортируют по нему. Бюджет/календарь/PDF — по датам/id, не затронуты.

## 5. Валидации (полный список — спецификация фронт-валидатора)

Все проверки считаются на черновике в памяти. Деление на «ошибка/варнинг» — для формулировки и приоритета в UI; сохранение блокирует ЛЮБАЯ незакрытая запись из разделов A–E. Переиспользуем чистые функции `validation.js` (`hotelWarnings/activityWarnings/transferWarnings/transferGroupWarnings/tripWarnings`, `sortVisits`) — они не лезут в БД. То, чего нет, — добавить.

### A. Города (новый валидатор; сейчас overlap живёт только в `CityVisitDialog`)
- **A1. Город без дат** — `start_datetime`/`end_datetime` обязательны для `kind='transit'`. Якоря `start`/`end` (даты null by design) — ПРОПУСКАТЬ, не считать ошибкой.
- **A2. Конец раньше начала** — `start > end` внутри города → ошибка.
- **A3. Пограничный день допустим с обеих сторон.** Города могут начинаться/заканчиваться в один день (стыковка) или в соседние дни (ночной перелёт). Нарушения:
  - **наслоение глубже пограничного дня** — пересечение интервалов больше чем на 1 пограничный день → варнинг;
  - **разрыв больше 1 дня** между соседними городами → варнинг.
  - (Заменяет жёсткий блок `CityVisitDialog.jsx:100`, который запрещал любое пересечение.)

### B. Отели (`hotelWarnings`, `validation.js:28`)
- **B1. Заезд раньше прибытия в город** — `check_in < visit.start` (есть, стр. 37).
- **B2. Выезд позже выезда из города** — `check_out > visit.end` (есть, стр. 39).
- **B3. Отель без города** — `city_visit_id` null/указывает на удалённый в песочнице город → сирота, «бронь без города».
- ~~Пересечение двух отелей в одном городе~~ — **НЕ проверяем** (решение Pavel; убрать ветку стр. 42-49).

### C. Активности (`activityWarnings`, `validation.js:53`)
- **C1. Начало раньше прибытия в город** — `start < visit.start` (есть, стр. 60).
- **C2. Конец позже выезда из города** — `end > visit.end` (есть, стр. 61).
- **C3. Активность без города** — как B3, сирота.

### D. Переезды (`transferWarnings`, `validation.js:65` + добавления)
Симметричные проверки обоих концов (B/C-стиль):
- **D1. Вылет позже выезда из города отправления** — `dep > fromVisit.end` (есть, стр. 80).
- **D2. Вылет раньше выезда из города отправления** — `dep < fromVisit.end` (НОВОЕ; симметрично D1).
- **D3. Прилёт раньше въезда в город назначения** — `arr < toVisit.start` (есть, стр. 84).
- **D4. Прилёт позже въезда в город назначения** — `arr > toVisit.start` (НОВОЕ; симметрично D3). Закрывает кейс A (Рим сдвинут на 13, билет на 16 — сейчас молчит, т.к. 16 внутри окна города).
- **D5. Трансфер между несоседними узлами** — концы `from/to_city_visit_id` не являются соседней парой в `(start_datetime, position)`-упорядоченном списке **узлов** (city + waypoint, см. §11; между ними есть третий узел) → варнинг «маршрут не сходится». НОВОЕ; сейчас не детектится. Соседство считаем **строго** по упорядоченному списку `(start, position)`, БЕЗ допуска на пограничный день (`position` детерминированно разрешает узлы с общим днём, §4a). **Включает кейс «назад во времени»** (`fromVisit.start > toVisit.start`) как частный случай — отдельная проверка не нужна. Перелёт со стыковкой = цепочка ОТДЕЛЬНЫХ трансферов между соседними узлами через waypoint (`city1 → wp1 → city2`), каждый трансфер между соседями — D5 одно правило на всё.
- **D6. Висячий конец** — `from`/`to` указывает на удалённый в песочнице узел (null) → сирота.

Группа `transferGroupWarnings` (`validation.js:97`, `MAX_TRANSFER_SEGMENTS`) — **удаляется.** Составной перелёт = цепочка отдельных трансферов между соседними узлами (через waypoint, §11), а не несколько сегментов в одной строке. На одну пару соседних узлов допустим только 1 трансфер (см. E3).

### E. Уровень трипа (последовательность; `tripWarnings`, `validation.js:136`)
- **E1. Нет переезда между соседними городами в разных местах** — для каждой соседней по `(start, position)` пары визитов с РАЗНОЙ идентичностью города и нет трансфера `from=A,to=B` → варнинг. **Критично:** вынести из рендера (`hasTransferBetween`, `TripView.jsx:787`) в `validation.js` как проход по соседям, иначе в песочнице её нет.
- **E2. Один и тот же город подряд** (Мадрид→Мадрид) — переезд НЕ требуется, E1 НЕ показывать. Различать по идентичности города (имя/external_city_id), НЕ по `city_visit_id`.
- **E3. Дубликат переезда** — на одну пару соседних узлов допустим **только 1 трансфер**; больше = варнинг «дубликат». Многосегментный перелёт = цепочка отдельных трансферов между соседними узлами через waypoint (§11), не несколько сегментов в одной строке. `MAX_TRANSFER_SEGMENTS` удаляется.
- ~~Разрыв >24ч (`tripWarnings` стр. 166)~~ — заменяется правилом A3 (разрыв >1 дня между городами). Старую ветку 24ч убрать/привести к A3.

## 6. Сохранение (upsert по id + явные удаления)

Выбран **upsert существующих по сохранённому id + insert новых + явный delete помеченных**. НЕ «снести весь трип и вставить заново».

**Почему не перезатирание (проверено по схеме dev):** `budget_expenses.source_id` — это **НЕ FK** (внешних ключей на события нет, есть только `trip_id`/`category_id`/`created_by`); связь «расход ↔ событие» держится логически триггером `sync_budget_expense` по паре `source_kind+source_id`. Поэтому при delete+reinsert событие получает **новый id** → триггер создаёт НОВЫЙ дефолтный расход, а любая ручная правка авто-расхода (сумма/категория) **теряется**. Это тихая потеря пользовательских данных. Документы (`documents` jsonb `{file_url,file_name}`, storage `attachments/{uid}/{name}` — `DocumentsField.jsx:57`) — файлы переживут, но любые ссылки на event id порвутся. При сохранении id ничего не рвётся.

**Почему не честный field-by-field diff:** он требует хранить исходный снимок и сравнивать поля каждой сущности — лишний код. Нам это не нужно: черновик песочницы УЖЕ содержит финальное состояние с реальными id существующих сущностей. Значит достаточно upsert (а не «вычислить что изменилось»).

Алгоритм Save (только когда валидаций нет):
1. **Удаления** — список «помечено к удалению» в песочнице (удаление отложенное, раздел 4). Явные DELETE в порядке `hotel_stays → activities → transfers → city_visits` (как `confirmDeleteCity`), чтобы триггеры `sync_budget_expense` отработали построчно корректно. NB FK-страховка асимметрична (проверено по схеме): `hotel_stays`/`activities` = `ON DELETE CASCADE`, но `transfers.from/to_city_visit_id` = `ON DELETE SET NULL` — на cascade для трансферов полагаться НЕЛЬЗЯ (останется сирота). Поэтому трансферы удаляем/репойнтим ВСЕГДА явно: оба конца к удаляемым узлам → DELETE; один конец выживает → это D6-сирота, блокирует сохранение, пока пользователь не переназначит/не удалит (§8.3). `trip_services` в batch НЕ участвуют (трип-уровень, не трогаем).
2. **Города** — upsert: существующие (`id` есть) `update`; новые (созданы в песочнице) `insert ... select id`, запомнить ремап старый-temp-id → новый-id для новых эвентов (как `ManualPlanner`/`copyTrip` `cityVisitIdMap`).
3. **Эвенты** (hotels/activities/transfers) — upsert: существующие по своему `id` (правки сохраняются на месте, бюджет/документы целы); новые — insert; `city_visit_id`/`from/to_city_visit_id` берутся из ремапа п.2 для новых городов.
4. **Бюджет** — НЕ полагаться на построчный шторм триггеров. После записи контента один проход синхронизации `budget_expenses` с актуальным набором событий трипа (меньше моргания сумм). Триггер `sync_budget_expense` оставить, цель — один пост-пересчёт.
5. Освободить лок (`editing_by = null`), инвалидация кэшей `TRIP_SHELL_KEY`/`TRIP_CONTENT_KEY` (`trip-data.js:11-17`).
6. Ошибка на любом шаге → откат/повтор, лок не освобождать, данные не терять. (По возможности обернуть в транзакцию/RPC, чтобы частичный сейв не оставил трип в полусостоянии.)

## 11. Тип узла `waypoint` (первоклассный узел маршрута)

Новый `kind='waypoint'` у `city_visits` — промежуточный узел маршрута (пересадка/тех. точка, которую пользователь хочет видеть как точку маршрута, напр. «провести день в Париже по пути»). NB (проверено по схеме): `city_visits.kind` имеет CHECK `IN ('transit','start','end')` — `'waypoint'` потребует миграцию `DROP/ADD CONSTRAINT` на prod И dev. Узел с одной датой = ровно «общий день» из §4a, поэтому `position` обслуживает waypoint без доработок.

- **Одна дата.** В отличие от `transit` (окно `[start, end]`), у waypoint **только одна дата** — это точка, не пребывание. Хранение: `start_datetime` = дата waypoint, `end_datetime = start_datetime` (либо null `end`, решить при реализации; рекомендуется `end = start`, чтобы существующая арифметика дат не падала). Без ночёвок.
- **Участвует в соседстве.** Waypoint — полноценный узел в отсортированном по датам списке. Все правила соседства (D5, E1, E2) работают по списку **узлов = city ∪ waypoint**. Маршрут `city1 → wp1 → wp2 → city2` = 3 соседние пары, для каждой нужен свой трансфер (E1).
- **Валидации узла (раздел A применяются с поправкой):** A1 — waypoint обязан иметь свою дату (без неё — ошибка); A2 неприменимо (нет интервала); A3 (пересечение/разрыв) — waypoint как точка участвует в проверке разрыва/порядка с соседями.
- **Что НЕ может waypoint:** к нему нельзя привязать отель (ночёвок нет); активность — спорно, по умолчанию запретить, решить при реализации. В выпадашке привязки отеля waypoint не показывать.

### Составной перелёт (стыковка) = цепочка отдельных трансферов через waypoint
Перелёт со стыковкой Мадрид→Цюрих→Рим моделируется как: city `Мадрид` → **трансфер** → waypoint `Цюрих` → **трансфер** → city `Рим`. Каждый сегмент — ОТДЕЛЬНАЯ строка `transfers` между соседними узлами; пересадка `Цюрих` — отдельный waypoint-узел со своей датой. **Единого «родительского» трансфера с массивом сегментов НЕТ** (вариант с `transfer_legs`/jsonb отвергнут Pavel: это вторая упорядоченная система параллельно маршруту — лишний источник истины). Один механизм: узлы + трансферы между соседями, D5/E1/E3 — единые правила и для прямых, и для составных перелётов. Если пересадку не хочется показывать как точку — она всё равно waypoint (просто транзитный); «провести там день» = у того же waypoint просто появляются активности/время, конвертация не нужна.

Следствие для текущего кода: `extraSegments` в `EventEditDialog.jsx:553` (доп-сегменты как отдельные строки transfer с ОДИНАКОВЫМИ концами + текст в адресах) — заменяется на «сегмент = трансфер между waypoint-узлами». Существующие данные с одинаковыми from/to и текстовой пересадкой — мигрировать (создать waypoint-узлы из текстовых пересадок) либо оставить как legacy с варнингом.

## 12. Этапность waypoint
Тип узла `waypoint` (§11) — **отдельный этап ПОСЛЕ** базового Edit Mode (разделы 1-10). В базовом Edit Mode заложить инварианты, чтобы потом не переделывать: (а) соседство/D5/E1 считать по `(start, position)`-списку узлов, готовому принять `kind='waypoint'`; (б) трансфер всегда между соседними узлами; (в) не завязывать модель транспорта жёстко на текущие `extraSegments` (`EventEditDialog.jsx:553`) — это временное решение, мигрирует в «трансферы между waypoint-узлами»; (г) миграция CHECK на `city_visits.kind` для `'waypoint'` (prod+dev) — отдельный шаг этой фазы, в базовом Edit Mode не нужен.
