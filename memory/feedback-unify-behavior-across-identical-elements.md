Если UI-элемент встречается в нескольких местах (карточки, кнопки, ряды, тогглы — «идентичные элементы»), то любое изменение его **поведения/вида/взаимодействия** применяется СРАЗУ КО ВСЕМ его экземплярам, через **одну общую реализацию** (компонент/хук), а не точечно в одном месте. Расхождение между идентичными элементами = баг и нарушение reuse-правила #6.

**Грабли (TRIP-140, 2026-06-27):** я поменял поведение карточек ОТЕЛЕЙ (клик = select, ссылка только по «Забронировать»), но карточки АКТИВНОСТЕЙ (`ViatorActivityList`) оставил со старым (клик по всей карточке = переход). Один и тот же визуальный элемент стал вести себя по-разному, и я об этом даже не предупредил. Pavel: «reuse и унификация — наши главные правила, сколько можно». Фикс: вынес общий `PartnerResultCard` (оболочка `.pcard` в app.css + единая логика взаимодействия) и перевёл оба списка (`Stay22HotelList` + `ViatorActivityList`) на него — теперь поведение физически одно.

**Эталон взаимодействия карточки-результата (PartnerResultCard):** клик по невыбранной → select; клик по уже выбранной → открыть ссылку (как кнопка); кнопка «Забронировать» → всегда открыть; hover → колбэк для синка карта↔список; Enter/Space = клик. Контент тела (рейтинг/цена/адрес) — слотами, поведение и шелл — общие.

**Why:** reuse-first/унификация — Hard rule #6; разное поведение у одинаковых на вид элементов нелогично для пользователя и я заставил Pavel ловить это вручную. Прямое усиление [[feedback-reuse-first-unification]].

**How to apply:**
1. ПЕРЕД изменением поведения/вида любого повторяющегося элемента — `grep` его собратьев (другие списки/карточки/кнопки того же паттерна). Не считать «правлю одну карточку» локальным.
2. Изменение для одного экземпляра = изменение для всех. Если общей реализации ещё нет — сначала вынести её (компонент/хук), потом менять в одном месте. Не форкать поведение копипастой.
3. Если расхождение ВСЁ ЖЕ нужно (элементы только похожи, но семантически разные) — объявить это явно в ответе ДО кода («меняю X у отелей, но НЕ у активностей, потому что…») и получить подтверждение, а не оставлять молча.
4. Закрывать UI-изменение строкой Reuse audit (правило #6) с явным перечислением, что ещё использует этот же элемент и что оно тоже обновлено.
