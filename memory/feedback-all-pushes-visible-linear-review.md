Pavel требует: **ВСЕ до единого пуши агента он должен видеть в Linear review.** Любой `git push`, после которого в Linear-треде не появился явный, актуальный указатель на PR и его свежее состояние — это провал. Нельзя оставлять «висящий где-то» пуш, которого Pavel не видит.

**Контекст инцидента (TRIP-100, 2026-06-26):** первый пуш открыл PR #164; на фидбэк Pavel я сделал второй коммit и `git push` в ту же ветку (PR обновился, новый PR не создавался — корректно по git), но **не отписал в тред**, что запушил апдейт. Linear-review показывал снапшот с первого пуша (старый «мобильный» дизайн), Pavel решил, что в PR висит старый дизайн и что мердж в `dev` накатит старое. На деле HEAD PR #164 уже содержал фикс — но с его точки зрения это был невидимый провал.

**Why:** Pavel ведёт ревью и мердж (агент НЕ мёрджит, особенно в dev/main — правило #12). Если он не видит свежий пуш в Linear, он либо мёрджит устаревшее, либо блокируется/теряет доверие. PR — единственная привязка работы к Linear-задаче (правило #5).

**How to apply:**
1. После КАЖДОГО `git push` (включая обновление уже существующего PR) — явным сообщением в треде: номер PR + ссылка, какой коммit/SHA теперь HEAD, что именно изменилось, и что мердж принесёт ИМЕННО это (не старое).
2. Обновляй тело PR при доработках, чтобы Linear-review отражал финальное состояние, а не первый снапшот.
3. Никогда не оставляй «голый» push без видимого в Linear PR-следа (это и есть правило #5, усиленное: видимость > факт пуша).
4. Если пушей несколько — каждый сопровождай отчётом; не полагайся на то, что Pavel сам найдёт обновление.

Связано с [[feedback-do-cleanups-immediately]] и git-правилом #5 в repo CLAUDE.md.
