ФАКТ (как работает СЕЙЧАС, после фикса 2026-06-05):

**Модель времени.** Все datetime сущностей (`hotel_stays.check_in/out/free_cancellation_until`, `transfers.start/end`, `activities.start/end`, `trip_services.pickup/dropoff`) хранятся как «настенное-как-UTC» (тип `timestamptz`, но `localToUtc` в src/lib/time.js пишет введённые цифры с суффиксом Z без сдвига). См. [[triplanio-localization]] и наивную модель таймлайна. Отдельной tz-колонки у сущностей НЕТ. Реальный пояс берётся из `city_visits.timezone` (hotel/activity → city_visit_id; transfer → from_city_visit_id для start; car_rental → `details.pickup_timezone`/`dropoff_timezone`, т.к. у trip_services нет связи с городом). Прод-здоровье: 121 city_visit, 0 null, 2='UTC' (ручные Париж/Леон в трипе «Франция», провал резолва), 119 реальных.

**Баг.** `get_pending_reminders(window_minutes)` (edge `getPendingReminders`, n8n каждые ~15 мин) сравнивал `entity.datetime BETWEEN now()+lead AND now()+lead+window`. `entity.datetime` = настенное-как-UTC, `now()` = настоящий UTC → расхождение ровно на offset города события. Мадрид (+2) → ремайндер на 2ч позже (событие 21:00, «за 24ч» приходил в 23:00), Ереван (+4) → на 4ч позже. Daily-функции `get_trips_*_tomorrow` (сравнение `::date = current_date+1`) НЕ ломались (дата-гранулярность) — их НЕ трогали.

**Фикс** (миграция `supabase/migrations/0020_reminder_tz_aware.sql`, задеплоено prod tizscxrpuopobgcxbekf + dev nydhzevdizkfaxdlikgc 2026-06-05). Хелпер `reminder_true_instant(ts, tz)`: если tz null/''/'UTC' → ts без конвертации (легаси + корректно для настоящих UTC-городов); иначе `(ts at time zone 'UTC') at time zone tz` = настоящий момент. Применён ко всем 7 типам в `get_pending_reminders`. Семантика (решение Pavel): «за N часов» = N часов до реального момента события В ПОЯСЕ ГОРОДА события (заселение 12:00 Мехико → ремайндер сегодня 12:00 Мехико). Дисплей/context = to_jsonb(...) не тронут, n8n форматирует настенное время как раньше.

ФИКС ФОРМУЛИРОВКИ «ЗАВТРА» (2026-06-12, prod+dev). Симптом: ремайндер активности/трансфера (lead = 4ч, всегда тот же день) писал «завтра» про событие через ~4ч. Корень: системный промпт ноды AI Agent в n8n-воркфлоу **TG Reminders** (`hDCVQcNdgEyKEvnY`) ХАРДКОДИЛ «событие, которое состоится завтра» + модель сама делала tz-математику (ненадёжно). Решение: вынес расчёт из модели в RPC. `get_pending_reminders` теперь добавляет в каждый `context` готовые поля: `event_instant_utc` (=reminder_true_instant), `starts_in_minutes`, `event_local_time` (HH24:MI настенное), `event_local_date`, `relative_day` (today/tomorrow/in_N_days, считается по дате в поясе события). Аддитивно (старый промпт игнорил). Lead-офсеты по типам: hotel_checkin 24ч, hotel_checkout 18ч, hotel_cancel 24ч, transfer_start 4ч, activity_start 4ч, car_pickup/dropoff 18ч. Промпт AI Agent переписан (2 поля Text+System Message, вручную в UI — MCP-апдейт рискует кредами): убран хардкод «завтра», модель использует relative_day/event_local_time/starts_in_minutes дословно, время показывает как настенное без конвертаций, now форсён `{{ $now.toUTC().toISO() }}`. ИСПРАВЛЕНО в **TG Chat Bot** (`mJ4QQUW7rrGh4tys`) 2026-06-12: живой системный промпт = code-блок Notion-страницы «🟢 TG Chat Bot — System Prompt [SOURCE OF TRUTH]» (`3702c9f1427e81d09707e33e938ac9f8`); страница «TG Chatbot Instructions» в Архиве — НЕ живая. Бот получает от `getTripByTelegramChatId` сырые строки таблиц (`select *`) → datetime = настенное-как-UTC + рядом `cityVisits.timezone`. Раздел «# Дата и время» врал («*_datetime в UTC, конвертируй UTC→локаль») → двойной сдвиг (19:00→21:00, перекат на завтра). Переписан на: datetime — настенное время места, показывать как есть без конвертации; timezone только для «сегодня/завтра»; now в UTC. Правка поля Text ноды AI Agent (`now: {{ $now }}` → `{{ $now.toUTC().toISO() }}`) — вручную в UI. Формат бота пока остался «плоский текст без Markdown» (в reminder включили legacy Markdown + parse_mode на ноде; для бота — отдельно, если попросят).
