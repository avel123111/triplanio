**КАРКАС (остаётся как есть):** трип держится на `city_visits` (город+даты). **РЕШЕНИЕ = отдельный экран Edit Mode (песочница):** пока юзер редактирует — НИЧЕГО не пишется в БД, всё в памяти/sessionStorage, всё вычисляется на лету. Экран = блоки городов с датами + переезды между (БЕЗ таймлайна). Можно двигать/добавлять/удалять города, править даты, править/добавлять/удалять события. Удаление города/эвента в песочнице = только пометка, физический DELETE на сохранении.

**ЗАМОРОЗКА МУТАЦИЙ (ТЗ §3a):** пока трип под редактированием (editing_by занят) — блокировать создание/изменение/удаление ВСЕХ эвентов (hotel/transfer/activity/service) из ВСЕХ точек входа: TripView (setHotelEdit/Choice, setTransferEdit/Choice, setActivityEdit, setService*, AddDayButton, Missing*Warning add-кнопки, openEventView→EventEditDialog delete, ServiceDialog, confirmDeleteCity) + прямые delete в HotelTimeline.jsx:20, ActivityList.jsx:27, TransferStrip.jsx:30, ServicesWidget. ВИД таймлайна НЕ ограничивать — только мутации. NB: TripView.isEditMode (инлайн-режим таймлайна) ≠ Edit Mode-песочница; инлайн тоже выключить на время песочницы.

**ГЕЙТ СОХРАНЕНИЯ:** пока есть хоть один варнинг/ошибка — «Сохранить» заблокирован. Чисто → diff-batch запись. Обоснование жёсткости (не мягкие варнинги): таймлайн `TripView` физически не отрисует произвольно несвязный трип (перелёт город7→город2, отель вне всех городов) → гейт «валидно на входе в таймлайн» = размен, чтобы не учить таймлайн рисовать хаос. Песочница без таймлайна — место, где промежуточный хаос ок (не сохраняется).
