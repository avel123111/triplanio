**★КАНОНИЧНАЯ ФОРМУЛИРОВКА (skeleton vs schedule) — суть всего редизайна:** СТРУКТУРА = скелет: явная упорядоченная цепочка узлов (stay/waypoint/anchor) + трансферы как РЁБРА между соседями; хранится явно (position + какое ребро какие 2 узла соединяет), НЕ выводится из дат → не рвётся ни при каких правках дат. ГИБКОСТЬ = расписание: даты пребываний и времена трансферов/событий, независимы, пересечения разрешены, блоков нет. Сегодняшняя жёсткость = скелет ЗАКОДИРОВАН В ДАТАХ (A.выезд=B.въезд, overlap запрещён, трансфер прибит к 2 городам по датам) → правка расписания калечит скелет. РЕШЕНИЕ = разделить скелет и расписание. ЛОГИКА/варнинги считаются ПО скелету (нет ребра между соседями→«нет трансфера»; время ребра вне окон соседей→подсказка) — надёжнее, чем по сортировке дат. Проверено на 3 кейсах Pavel (сдвиг выезда A / въезда B / города B): расписание течёт, ребро A→B постоянно, скелет цел. Реордер маршрута = структурная правка (отдельный жест), не правка дат.

**★КРИТИЧНО — модель времени (проверено в коде 2026-05-31):** трип-таймлайн использует НАИВНОЕ wall-clock, city.timezone в РЕНДЕРЕ ИГНОРИРУЕТСЯ. time.js дословно: ianaTz-параметры не используются, "CityVisit.timezone is NOT used for time display anywhere on the trip timeline/dialogs". check-in "15:00" хранится как ...T15:00:00Z (цифры стенки, без оффсета) и так рисуется (naive-time.js parseNaive). tz РЕАЛЬНО живёт только в: валидации (validation.js setZone(visit.timezone) для day-сравнений), погоде/гео, и transfer.details.pickup/dropoff_timezone. → СЛЕДСТВИЕ: отвязка hotel/activity/transfer от города НЕ ломает отображение времени (оно self-contained); у каждой сущности есть свои lat/long → tz резолвится (resolveTimezoneFromCoords). Прежняя посылка «tz берётся с города, поэтому нельзя отвязать» — НЕВЕРНА для рендера. Расхождение naive(рендер) vs tz(валидация) = одна из причин «таймлайн живёт своей жизнью».

**★УТОЧНЕНИЕ «самодостаточности» (после пушбэка Pavel):** слово «самодостаточный» НЕ применять к отелю/активности/трансферу — они ДЕТИ города, членство реальное и ОСТАЁТСЯ (поэтому карточка отеля в CityHero, поэтому мы всегда знаем, есть ли в городе бронь). Самодостаточность относится ТОЛЬКО к: (1) у события своё время+tz (tz из собственного адреса события, иначе дефолт tz города) — чтобы ремайндер/рендер не зависели от города; (2) порядок городов = хранимый position. Членство hotel→город задаётся тем, ПОД каким городом добавили (кнопка в герое), отдельно город не спрашиваем (в отличие от car rental). ПЕРЕМЕЩЕНИЕ ГОРОДА: член (отель/активность) по умолчанию ЕДЕТ вместе с городом на ту же дельту (превью+undo, позиция внутри пребывания сохраняется); исключение — подтверждённая/невозвратная бронь: не двигаем молча, помечаем «двигать вместе / оставить даты». Данные НИКОГДА не теряются. Трансфер — член двух городов-концов, двигается соответствующий конец. ИТОГ: гибкость = связь отель↔город из «жёсткого date-констрейнта (блок+каскад)» → в «членство: едет с городом по умолчанию + максимум варнинг». Это вся разница с текущим.

**ГОРОД играет 5 ролей; жёсткие только #1-3:** (1) порядок маршрута — из дат sortVisits, нет position [ХАРД]; (2) запрет наслоения — CityVisitDialog canSave блок [ХАРД]; (3) трансфер к 2 городам по id, составной = N строк с одинак. краями [ХАРД]; (4) валидация-сосед/варнинги [софт, но часть выродилась в блок #2]; (5) семантич. дом + группировка под CityHero (city_visit_id NOT NULL) + tz/погода [софт, не трогать]. Гибкость = понизить #1-3 из constraint в reference: #1→хранимый position; #2→снять блок, overlap=варнинг; #3→трансфер между СОСЕДНИМИ узлами, составной=цепочка через промежуточный узел со своими коорд/tz; #4→варнинги остаются, им нужны только порядок(position)+самодостаточные сущности, НЕ жёсткие id; #5→город=дом-группировка, tz из coords сущности.

**РАЗВИЛКА ВРЕМЕНИ — РАЗРЕШЕНА требованием ремайндеров (2026-05-31):** наивное-везде = НЕ вариант, это и есть причина, почему ремайндеры сломаны. ДОКАЗАТЕЛЬСТВО из кода: getDailyReminders шапка — «UTC date arithmetic, datetimes stored as naive wall-clock-as-UTC, plain ::date» → «завтра» по UTC-дате, не по локальной. getPendingReminders «за 24ч до чекина» шлёт за 24ч до 15:00 UTC, хотя 15:00 локальные = другой инстант → ошибка на оффсет города на КАЖДОМ событии. ЕДИНСТВЕННЫЙ корректный ремайндер — аренда авто: хранит pickup_at_local + pickup_timezone, «legacy without TZ are skipped» (команда уже решила это для авто верно). РЕШЕНИЕ ВЫНУЖДЕНО: каждое timestamp-событие (чекин/аут, активность, вылет/прилёт) несёт ЛОКАЛЬНОЕ время + свою IANA-tz. Координаты у всех уже есть (hotel.lat/long, activity.location_lat/long, transfer.from/to_lat/long) + resolveTimezoneFromCoords. КЛЮЧ: per-event tz = ТО ЖЕ действие, что развязывает событие от города. «Починить ремайндеры» и «дать гибкость» = одна операция. UX «ввёл 15:00 — видишь 15:00» сохраняется (дисплей = инстант в tz события). Город опускается до: дефолт tz при создании + fallback без координат (иначе скип, как авто) + группировка/день.
ФОРМА ХРАНЕНИЯ (осталось выбрать, на логику не влияет): (A, рекоменд.) *_at_local + *_timezone рядом — как у авто, не ломает наивный ввод, чинит бэкофиллом tz по координатам; (B) канонич. timestamptz истинный инстант + колонка timezone — одно поле, но переписать localToUtc + аккуратная миграция наивных значений. **Решения Pavel:**
- Полный передел: маршрут = единый упорядоченный поток узлов с ХРАНИМЫМ `position` (порядок ≠ даты). Даты — мягкий атрибут (двигаются свободно).
- Типы узлов: `stay` (город с ночёвками, бывш. transit) / `anchor` (start/end) / `waypoint` (пересадка/тех. точка: место+время, без ночёвок по умолчанию).
- Составной трансфер = вариант A (waypoint): трансферы между СОСЕДНИМИ узлами (Мадрид→Париж(wp)→Рим). Технический layover рендерится СВЁРНУТО внутри карточки трансфера; «провести день» → промоут waypoint в город.
- ★ГЛАВНЫЙ ПРИНЦИП (исправлено после пушбэка Pavel): ИСТОЧНИК ИСТИНЫ = ВРЕМЯ, не город. Город (stay) = интервал [прилёт,выезд]+место, просто полоса-контекст. Активности/отели/трансферы = самостоятельные время-объекты (своё время+место), таймлайн рисует их СТРОГО по их времени. «Какому городу принадлежит момент» — ВЫЧИСЛЯЕТСЯ (какой stay-интервал накрывает), не хранится. Ничто не «сидит внутри» города.
- СХЕМА (живая, dev nydhzevdizkfaxdlikgc, 2026-05-31): hotel_stays.city_visit_id и activities.city_visit_id = NOT NULL (жёсткая ГРУППИРОВКА к городу; check_in/out, start/end nullable; у отеля/активности свои lat/long/address). transfers.from/to_city_visit_id = УЖЕ nullable. position нет нигде. Порядок сегодня = sortVisits по start_datetime + ранг kind (validation.js:174).
- ПОРЯДОК (исправлено, retract «тай-брейкер»): макро-порядок городов = ХРАНИМЫЙ position, ВСЕГДА (не из дат — даты nullable: якоря/новые/passthrough). Даты порядок НЕ определяют, только расписание. Для порядка время не используется; единственная дата-ось — вертикаль CityHero внутри дня = по ПРИЛЁТУ (start_datetime). Если position рассинхронен с датами (обратный порядок) → рендер ПО position (детерминированно), несоответствие = неблокирующий варнинг + фикс [сорт.по датам]/[поменять]/[править]. Никогда не пересортировывать молча, не блокировать.
- КОНФЛИКТЫ считаются попарно на соседних по position: инверсия дат P(i).start>P(i+1).start; пересечение интервалов; отель/активность вне окна города; вылет/прилёт трансфера вне окон соседей; нет трансфера между соседями в разных местах. Каждый — варнинг.
- ПРИВЯЗКА к городу (city_visit_id) ОСТАЁТСЯ как группировка (под каким героем показывать). «Мягкость» = ПОВЕДЕНИЕ, НЕ nullable: ноль авто-мутаций дат брони, ноль блоков, размещение по времени, валидация = варнинги. nullable city_visit_id рассматриваем ТОЛЬКО для «оставить бронь без города» при удалении города (решение дизайн-дока). (Прежние тезисы «nullable advisory тег» и «position=тай-брейкер» — отменены.)
- ОТЕЛИ при изменении дат города (броня священна, авто-мутация/подрезка/блок ЗАПРЕЩЕНЫ): сжал→«check-out позже выезда» варнинг (есть); расширил→gap «нет отеля» (есть); сдвинул→опция «двигать брони с городом» одним дельтой (наслоение сохраняется), превью+undo; неск.наслоенных→независимые интервалы, overlap уже варнится; удалил город→диалог [перепривязать к соседу]/[без города]/[удалить с бронями].
- Трансфер from/to-город МЯГКИЕ (уже nullable): группировка + варнинг «нет переезда»; связность по месту+времени соседних stay.
- Overlap дат / любые нестыковки → только НЕблокирующий варнинг + быстрый фикс. Не каскадить по констрейнту.
- EDIT-СЕМАНТИКА (ключ к гибкости): любой edit ВСЕГДА проходит, ничего не теряется/не блокируется. Двинул город → выбор «только город / город+брони (превью+undo)». Перестановка городов = своп временных слотов (превью): город со своими бронями едет тем же сдвигом; остаток нестыковок = варнинг. Пример: city2 14-15(+hotelB)+city3 15-17(+hotelC), своп → city3 14-16(+hotelC), city2 16-17(+hotelB) — не ломается.
- Intraday рендер: hero ставится по времени ПРИЛЁТА; день = единый поток событий, слитый и отсортированный по времени (сейчас TripView.jsx:901-924 рисует hero сверху потом события — меняем на time-merged). Неск. городов в 1 день = неск. arrival-блоков по времени прилёта; активности — на своём времени, не «под hero».
- Выкат: prod = только Pavel, поэтому катим И dev И main одновременно; передел в ОДНОЙ схеме, выкатывать слоями за флагом, прод не должен падать между шагами.
