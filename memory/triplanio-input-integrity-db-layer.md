★PR в dev 2026-07-09 (TRIP-169, рефокус). Задача исходно просила «единый edge-валидатор тел (zod/valibot)». Аудит показал: это неверный центр тяжести для нашего стека. Прямой PostgREST CRUD — доминирующий путь записи (~60 мест, ~10 таблиц), edge покрывал бы 1 путь из 3; авторизацию уже держит ярусная RLS [[triplanio-security-tiers-architecture]]; enum'ы уже закрыты 76 CHECK'ами (baseline). Единственный реальный пробел — **нет кэпов длины строк и диапазонов чисел**: все пользовательские колонки безлимитный `text`, ни одного `CHECK (amount >= 0)`. В модели «БД и есть API» инвариант формы обязан жить в БД (единственный слой, покрывающий все три пути сразу), поэтому TRIP-169 сделан как DB-слой, НЕ edge-zod. Отклонены как оверинжиниринг: единая edge-граница с выпилом CRUD (сменa парадигмы, выбрасывает RLS), zod-зависимость, кодоген схемы во все слои.

**Что в миграции `20260709121734_trip169_input_integrity_constraints.sql`:**
- 3 домена «safe by construction» для БУДУЩИХ колонок: `public.short_text` (≤300), `public.long_text` (≤10000), `public.url_text` (≤2048). Существующие колонки НЕ конвертировали в домены (вью-зависимости ломают `ALTER COLUMN TYPE`) — им даны эквивалентные `char_length` CHECK'и.
- Кэпы длины на ~55 пользовательских строк (short 300 / long 10000 / url 2048 / email 320 / currency·country_code 8).
- Неотрицательность чисел: `price`/`original_amount`/`position`/`order_index` → `col IS NULL OR col >= 0`.
- **Числа проверяются на `>= 0`, а НЕ `> 0`** (важно): на prod 63 живых `budget_expenses.original_amount = 0` — это легитимный синк из броней с нулевой ценой (триггер `sync_budget_expense()`), `> 0` уронил бы миграцию и ломал синк. Фронт (`validation.js`) требует `>0` только для РУЧНОЙ траты — это UX-слой, не бэкстоп.
- Метаданные-only, откат = `DROP CONSTRAINT`/`DROP DOMAIN`. Замер dev+prod подтвердил ноль нарушений на существующих данных (макс строка — `trips.cover_image_url=532` < 2048).

**CI-гард 2g `scripts/ci/check-column-caps.mjs`** (в `checks.yml`, по образцу `check-destructive-migrations`): падает, если новая миграция добавляет сырую `text`-колонку без кэпа (домен или `char_length` CHECK). Escape-маркер `-- caps-guard: allow-uncapped — <reason>`. Так «добавили поле без гварда» невозможно по построению — на уровне PR. Существующая схема грандфадерится (гард смотрит только added-строки диффа).

Нарушение CHECK PostgREST отдаёт как **400 (23514)**, а не 500 — попутно улучшает контракт ошибок на write-пути. `loadStateClassify` (22P02→404 на ЧТЕНИИ) ортогонален и остаётся. Отложено отдельными мелкими PR: `copyTrip` `...spread` → явный whitelist колонок (связанность с физической схемой, не дыра); нормализация edge error-контракта к `{error,code}`. Усиливает [[feedback-design-for-scale-not-now]] (сделали правильный слой, а не большой).
