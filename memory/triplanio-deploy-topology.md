Топология деплоя (обновлено 2026-06-29, TRIP-134 Ф3 — последовательная цепочка бек→фронт):

- **Агент НЕ деплоит сам и не предлагает ручной деплой** — всё через CI/CD (merge→GitHub Actions). См. [[feedback-no-manual-deploy-cicd-only]].
- **Один воркфлоу `.github/workflows/supabase-deploy.yml` (имя «Deploy (backend → frontend)») катит И бек, И фронт одной упорядоченной цепочкой** (TRIP-134 Ф3, с 2026-06-29, на dev): `gate + detect → migrate → deploy(функции) → vercel(фронт) → notify(Telegram)`. Порядок через `needs` + result-aware `if: !cancelled() && gate/detect==success && upstream (success||skipped)`. **Функция не обгоняет миграцию, фронт не выходит раньше бека**; падение останавливает цепочку до фронта (фронт остаётся на последней рабочей сборке). **Front-only fast-path:** если `supabase/**` не тронут — migrate/deploy скипаются, vercel бежит сразу после гейта. Триггер push в dev/main **без paths-фильтра** (фронт должен катиться на любой push). Verified зелёным на dev (run 28402056074): dev.triplanio.com обновился, Telegram пришёл.
- **Фронт (Vite-приложение) — теперь через CI по Vercel-токену, НЕ через Git-интеграцию Vercel** (TRIP-134 Ф3). Job `vercel`: `vercel pull/build/deploy --prebuilt` (preview на dev / `--prod` на main), токен из `VERCEL_TOKEN` (+`VERCEL_ORG_ID`=`team_…`, `VERCEL_PROJECT_ID`). CLI-деплой внутри чекаута на ветке dev/main сам навешивает бранч/прод-домен (dev.triplanio.com / triplanio.com) — доп. `vercel alias` НЕ нужен (проверено на dev). **Git-автодеплой Vercel для dev/main ВЫКЛЮЧЕН** через Ignored Build Step (`if dev||main → exit 0 (skip), else exit 1 (build)`) — preview на ветках-задачах остаётся. Бонус: деплой от токена обходит боль Hobby с коммитами не-владельца ([[triplanio-vercel-hobby-blocks-collaborator-commits]]). ⏳ prod-домен (triplanio.com через `--prod`) ждёт первого мерджа dev→main для подтверждения.
- **Видимость — Telegram-notify** (TRIP-134 Ф1): финальный job `notify` (`if: always()`) шлёт один пинг на success И failure с результатом каждого job + ссылкой на run, через ops-бота (секреты `TELEGRAM_OPS_BOT_TOKEN`/`TELEGRAM_OPS_CHAT_ID`; без них graceful skip).
- **PR-гарды в `checks.yml`** (TRIP-134 Ф2, job `guards`, только на pull_request): 2a нумерация миграций (dup/out-of-order/ручной круглый стамп), 2b деструктив (DROP/RENAME/ALTER…DROP без маркера `-- ddl-guard: allow-destructive`), 2c verify_jwt новой внешней функции, 2d i18n (дубль ключей + новые хардкод-строки в diff, подавление `// i18n-ignore`). Логика в `scripts/ci/*.mjs`. Блокируют плохой PR, рантайм не трогают.
- **Supabase edge-функции — АВТО через GitHub Actions** (TRIP-73, с 2026-06-25). Мердж в `dev` → деплой всех функций в Supabase **dev** (`nydhzevdizkfaxdlikgc`); мердж в `main` → в **prod** (`tizscxrpuopobgcxbekf`). `verify_jwt` только из `config.toml` + ассерт через Management API. Подробности: [[triplanio-cicd-github-actions]].
- **Supabase миграции — АВТО через GitHub Actions** (TRIP-68 Ф3, с 2026-06-25, после reconcile истории). Тот же воркфлоу `.github/workflows/supabase-deploy.yml`, job `migrate`: merge→`dev` → `supabase db push` в dev; merge→`main` → в prod; триггер по `supabase/migrations/**`; `detect`-job разводит functions-only vs migrations-only, чтобы не было лишних передеплоев. История = единый baseline `20260625120000_baseline.sql`; репо↔dev↔prod журналы идентичны. **Новые миграции ТОЛЬКО `supabase migration new` (таймстамп); `00NN` запрещены.** Требует секреты `SUPABASE_DB_URL_DEV`/`_PROD`. См. [[triplanio-migration-naming-drift]].
- **⚠️ `SUPABASE_DB_URL_*` обязаны указывать на SESSION-пулер (порт 5432), НЕ транзакционный (6543)** (TRIP-134, 2026-06-29). Свежий Supabase CLI (`db push`) делает внутренний `migration list` через ИМЕНОВАННЫЕ prepared statements; на транзакционном пулере они детерминированно коллизятся на переиспользованном пуловом бэкенде → `ERROR: prepared statement "lrupsc_1_0" already exists (SQLSTATE 42P05)`. Триггернула это Ф5 (re-stamp гоняет `migration list` перед `db push`, «отравляя» бэкенд) — до неё один `db push` за прогон обычно попадал на чистый бэкенд, поэтому «раньше работало». Ретрай НЕ помогает (все попытки летят на тот же тёплый бэкенд — это не транзиентный сбой). Фикс = строка подключения на session-пулер (5432): там у каждого коннекта свой бэкенд, prepared statements живут. CLI запинен (`version: 2.108.0`, не `latest`), чтобы поведение не дрейфовало. Прямое подключение тоже годится по семантике, но IPv6-only → не ходит с GitHub-раннеров без платного IPv4. Биллинг от смены режима пулера НЕ меняется. **DEV переведён 2026-06-29; PROD-секрет надо перевести так же до первого мерджа dev→main.**
- **Ручной деплой = аварийный фолбэк, только Pavel** (не агент): хотфикс без мерджа. После — `workflow_dispatch`, чтобы runtime == git. ⚠️ Ручной MCP/CLI-деплой функции сбрасывает `verify_jwt=true` — для pinned-false передавать `false` явно.
