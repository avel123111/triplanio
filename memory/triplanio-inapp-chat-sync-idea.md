Pavel рассматривает идею сделать InApp групповой чат СИНХРОННЫМ: приложение отправляет запрос, ждёт ответ, показывает состояние «Triplanio думает». **Решение 2026-05-30: приложение пока НЕ трогаем, вернуться к идее позже.**

**Текущая схема (async):** приложение пишет сообщение юзера в `chat_messages` (Realtime разносит всем) + зовёт edge-функцию `callTriplanioAi` (проверяет членство в трипе, собирает последние 20 сообщений, шлёт payload в n8n webhook `/group-chat` с N8N-JWT, отвечает приложению `{ok:true}` сразу). n8n (`InApp Group Chat Bot`: AI Agent2 + Gemini + `getTripById`) формирует ответ → POST в `triplanioAiReply` → вставка в `chat_messages` от бота (info@triplanio.com) → Realtime разносит всем.

**Главный вывод анализа:** синхронность НЕ нужна для UX «думает». Это групповой чат — ответ бота всё равно обязан дойти до ВСЕХ участников через `chat_messages`+Realtime, поэтому sync не убирает запись в БД, а добавляет второй путь + связывает таймауты.

**Риски sync:** таймауты по всей цепочке app→callTriplanioAi→n8n(ждёт весь workflow)→Gemini+getTripById→ответ (LLM 3–15с+, прокси Railway, edge-лимиты); дубли при ретраях (нужна идемпотентность); удержание соединений/исполнений; двойная доставка если оставить и `triplanioAiReply`-вставку, и возврат в теле.

**Рекомендация — Путь 1 (async + UX):** оптимистичный «Triplanio думает» на отправке, снимается приходом ответа по Realtime (+ фолбэк по таймауту); realtime-сигнал «печатает» для всех (флаг ai_pending/broadcast); ветка ошибки в n8n — при сбое писать фолбэк-сообщение бота (закрывает текущую слабость: при падении n8n юзер сейчас не получает ничего молча). Низкий риск, бэкенд почти не трогаем.
