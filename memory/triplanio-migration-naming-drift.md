**Симптом:** Supabase Git-интеграция (авто-деплой миграций на push в main) падает КАЖДЫЙ ран на обоих проектах с `Remote migration versions not found in local migrations directory` (лог: clone repo → connect db → эта ошибка). В UI ранов «MIGRATIONS: FAILED, 1 SKIPPED, 2 PENDING». Идёт с ~05 июн 2026.

**Первопричина (подтверждено 2026-06-07 через Supabase MCP):** разные схемы версионирования.
- Репо `supabase/migrations/`: 23 файла, имена `00NN_name.sql` (`0003`…`0023`), есть дубли номеров (`0006`×2, `0020`×2).
- БД хранят историю в `supabase_migrations.schema_migrations` таймстампами `2026…` (14 цифр). PROD `tizscxrpuopobgcxbekf` = 50 версий, все таймстампы; DEV `nydhzevdizkfaxdlikgc` = 33 версии, все таймстампы.
- Пересечение версий репо↔БД = **0**. Supabase матчит файл↔версию по числовому префиксу имени → `0003` ≠ `20260526…` → все серверные версии «нет локально» → abort.
- Репо (23) ≠ PROD (50) ≠ DEV (33): половина миграций накатана напрямую через MCP `apply_migration`/Studio и в репо файлами не попадала (напр. ранние RLS-debug версии — только на проде). Файлы `00NN` — неполный параллельный ручной мираж.

**Почему числа разные (50/33/23):** `schema_migrations` = ЖУРНАЛ применённых миграций, НЕ инвентарь схемы. PROD (50) — первая «грязная» история с ~11 RLS-debug пробами 26 мая + insurance + edit-lock(5) + cascade (часть накатана руками MCP). DEV (33) — поднят позже с чистого старта, без debug-возни; часть изменений приехала мимо журнала. РЕПО (23) — неполный ручной срез (с 0003, без initial/rls и последних 3 AI, дубли 0006/0020). ⚠️ Журнал ≠ реальная схема: edit-mode объекты (5 функций acquire/release/heartbeat_trip_lock+save_trip_edit/structure, trips.editing_by/since, city_visits position) есть на ОБОИХ проектах одинаково, хотя в журнале dev этих миграций нет (проверено 2026-06-07). Так что dev НЕ «отстаёт» по схеме — расходятся только летописи.

**Не ломает приложение** — БД актуальны (миграции накатываются вручную, см. [[triplanio-deploy-topology]]). Падает только авто-синхрон; фейлы = шум + маскируют реальные сбои.

**Варианты фикса (НЕ выбран, ждёт решения Pavel):**
- A) Отключить Supabase Git-интеграцию, продолжать накат вручную (статус-кво воркфлоу). Дёшево, дрейф остаётся.
- B) Принять Git-managed миграции: переименовать/перегенерить файлы под таймстампы реальных версий БД (`supabase db pull` + `migration repair --status applied` на КАЖДОМ проекте отдельно, prod/dev разные истории). Правильно, но трудоёмко и рискованно на проде. Отдельная задача.
- C) Оставить как есть (фейлы косметические).

⚠️ Любая правка `schema_migrations` на проде — высокий риск, только с подтверждением.
