★FK-ДРЕЙФ ВЫПРАВЛЕН 2026-06-23: `users_id_fkey` (public.users→auth.users) на PROD был `NO ACTION` (дрейф), на dev+Notion-доке = `ON DELETE CASCADE` (каноничен — на нём построена вся логика «нейтрализуем на месте, т.к. hard-delete тянет cascade в public.users→23503»). Выправил prod на CASCADE через MCP. Для рабочего флоу нерелевантно (auth.users никогда не удаляется), но дрейф убран. NB: НЕ путать сжатую формулировку «удалить auth.users» — auth.users НЕ удаляется.
★PII-долг снапшотов имени (всплыл при unify чата/доков [[resolveAuthor]]): `anonymize_my_account` зануляет `trip_members.user_full_name`, но НЕ `chat_messages.user_full_name` (давний пробел) и НЕ `trip_documents.created_by_name` (новая колонка 0062) → реальное имя удалённого остаётся в БД (не отображается — resolveAuthor показывает is_deleted раньше снапшота, но PII лежит at-rest). Перед деплоем 0062 добавить в anonymize: `update chat_messages set user_full_name=null where user_id=p_user_id` + `update trip_documents set created_by_name=null where created_by=p_user_id`. Обновить Notion-таблицу.

★ФИКС-2 2026-06-22: hard-delete `auth.users` НЕВОЗМОЖЕН — `chat_messages.user_id` (и др. сохраняемый контент) ссылается на `auth.users(id)` FK NO ACTION → `auth.admin.deleteUser` падал 23503 («Database error deleting user», 500). РЕШЕНИЕ (Pavel заранее согласовал фоллбэк «банить+затирать»): нейтрализуем auth ВНУТРИ RPC, в той же транзакции — `delete auth.sessions`, `delete auth.identities` (чтобы Apple/Google provider-sub не матчил→можно перерегистрироваться заново), `update auth.users set email=плейсхолдер` (БАН НЕ ставим — убрали 2026-06-22: он ничего не даёт и пугал блокировкой повторной регистрации; вход и так невозможен без identities/пароля/сессий, а re-signup создаёт новую строку). Цепочка блока hard-delete: public.users.id→auth.users ON DELETE CASCADE (users_id_fkey), а на public.users висит контент → 23503. chat_messages.user_id ссылается на public.users (НЕ auth). Email удалённого НЕ отдаём наружу — проекции get_trip_participant_profiles+resolveProfiles гасят его при deleted_at. Edge БОЛЬШЕ НЕ зовёт deleteUser (убрали) → нет частичного состояния. Проверено: security-definer RPC имеет права на запись в auth-схему. Стек: RPC всё делает атомарно, edge только вызывает RPC и маппит code. Edge dev v15 / prod v18.

★ФИКС-1 2026-06-22: `users.email` оказался NOT NULL+UNIQUE → `UPDATE email=null` валил RPC (500, транзакция откатывалась, deleted_at оставался null). **Сменили модель на soft-delete + анонимизация** (hard-delete невозможен: ломает совместный контент + теряет финансы; сторы Apple 5.1.1(v)/Google + GDPR допускают анонимизацию и хранение платёжных записей).

Согласованный план:
- Серверная RPC/edge в ТРАНЗАКЦИИ: (1) гард активной подписки (server = source of truth; фронт-предпроверка остаётся ради UX); (2) DELETE только `chat_reads`, `notifications`, `telegram_link_tokens`, `telegram_reminder_logs`, `trip_telegram_integrations`; (3) обезличить `public.users`: email→null (освобождает email для повторной регистрации), full_name→null, avatar_url→null, выставить `deleted_at`+статус; (4) удалить `auth.users` (вход пропадает).
- НЕ трогаем (общий контент/деньги): activities, budget_categories, budget_expenses, city_visits, hotel_stays, transfers, trip_services, chat_messages, trip_members, trip_documents, partner_clicks, trip_subscriptions. Обезличенного участника в совместных трипах ОСТАВЛЯЕМ как есть (роль не понижаем).
- Stripe: маппинг `stripe_customer_id`+`trip_subscriptions` в нашей БД ХРАНИМ (поздний вебхук возврата/диспута должен сматчиться). Сам Customer в Stripe оставляем (законно, процессор под DPA); опц. удаление Customer через API — отдельное решение, по умолчанию НЕ трогаем.

Гард подписки сейчас: edge блокирует если `trip_subscriptions.type ∈ (pro_monthly,pro_yearly) AND status='active'`; разовый `pro_trip` НЕ блокирует. Фронт: `plan.plan==='pro' && !plan.cancelled`.

Имя/аватар удалённого: НЕ хранить переведённый текст в БД. Признак = `deleted_at` (НЕ инферить по пустому имени — на dev 2/13 живых юзеров с full_name='', заведены напрямую/без имени). FE детектит `deleted_at` и подставляет `t()`-ключ + фикс. нейтральный аватар. Поле `deleted_at`/статус в `public.users` НАДО ДОБАВИТЬ (сейчас колонок: id,email,full_name,avatar_url,language,theme,notify_*,subscription_status,subscription_end_date,created_at,updated_at,stripe_customer_id,entitlement_synced_at — deleted_at/status НЕТ). `deleted_at` надо прокинуть в проекцию профиля для FE (resolveProfiles edge / get_trip_participant_profiles RPC) — проверить их select-список.
