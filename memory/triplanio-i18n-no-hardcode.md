# Triplanio: не хардкодить пользовательские строки

Pavel жёстко указал (2026-05-31): нельзя хардкодить текст «втупую под окружение», не думая о масштабируемости. Приложение мультиязычное — **en / es / ru**, языки будут добавляться.

**Why:** хардкод-строки не масштабируются, ломают i18n, добавление языка превращается в правку во многих местах. Это нарушение project-инструкций (масштабируемость, анализ влияния на проект).

**How to apply:**
- Фронт: использовать `t('...')` + locale-файлы `src/lib/i18n/locales/{en,es,ru}/*.js`. Добавляя строку — добавить во ВСЕ три локали. (Готовый пример — `TelegramAssistantPanel.jsx` использует `t()`.)
- **Важно:** `SettingsLens.jsx` (и, видимо, ряд `*Lens`/redesign-экранов) **захардкожены по-русски целиком** — это технический долг, отдельная задача «перевести лайнсы на t()». Если правишь там — матчишь существующий стиль, но помни про долг и предлагай локализацию.
- Бот/сервер-сайд (Telegram): локализацию держать в edge-функции, а не в n8n. `telegramWebhook` возвращает готовый `message` по языку (из `users.language` по user_id токена; для no-token/welcome — из Telegram `language_code`, фолбэк `en`), n8n шлёт `{{ $json.message }}`. Добавить язык = дописать блок в мапу `T` в одной функции. Связано: [[triplanio-telegram-multilink]].
- Общий принцип: перед «простым» решением подумать, как оно влияет на проект (i18n, реюз, дублирование, добавление новых сущностей/языков). Не делать первое попавшееся.
