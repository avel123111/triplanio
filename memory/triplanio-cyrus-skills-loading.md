**★ОБНОВЛЕНИЕ 2026-07-03 (TRIP-179) — `.claude/skills/` ТЕПЕРЬ ГРУЗИТСЯ, но из ветки активного клона (`main`), а не из ворктри/`dev`. Это ОТМЕНЯЕТ смоук TRIP-22 «skills не автоподгружаются» (был на старой версии Cyrus).** В свежей сессии в списке Skill-инструментов реально видны `ecc-*`, `design-*`, `engineering-*`, `web-quality-*`, `agent-skills-*`, `ui-ux-pro-max`, `example-frontend-design` + `cyrus-skills:*`. Механизм (по исходникам `cyrus-edge-worker`): allowlist сессии = **управляемые скоупы (`scope.json`) ∪ репо-локальные `.claude/skills/*`**, где репо-локальные читаются `resolveSkillRepoPaths` из **пути активного клона `/data/.cyrus/repos/triplanio`**, а тот стоит на ветке **`main`** (НЕ `baseBranch=dev`, НЕ ворктри задачи). Этот список Cyrus отдаёт в SDK как `skills: string[]` (context-filter; файлы на диске остаются, но невключённые скрыты). **Следствие: новый скилл в `.claude/skills/` становится вызываемым ТОЛЬКО после того, как доедет до `main`.** Кейс TRIP-179: `marketing-*` (45 шт, вендор TRIP-163/PR #329) влиты в `dev`, но НЕ в `main` → `origin/main:.claude/skills` содержит 0 marketing-папок → `Skill(marketing-product-marketing)` падает «not in this session's skills allowlist», хотя ворктри (от dev) их физически содержит. Фикс = мердж `dev→main` (делает Pavel) + обновить клон до нового main; проверка `git -C /data/.cyrus/repos/triplanio ls-tree -d --name-only origin/main:.claude/skills | grep -c marketing` → должно стать 45. Railway/env для этого править НЕ нужно (Cyrus не берёт SDK-опцию `skills` из env). Плагинные скиллы (`cyrus-skills:*`, user-skills) резолвит `SkillsPluginResolver` из `~/.cyrus/{cyrus-skills-plugin,user-skills-plugin}`. Rule 15 в CLAUDE.md (читать SKILL.md ридом) остаётся валидным фолбэком для скиллов, которых ещё нет на `main`.

**Сетап Pavel: self-hosted Cyrus БЕЗ веб-интерфейса** (OSS cyrus-процесс на Railway, не продукт app.atcyrus.com). Прямая цитата: «у меня нет дашборда у меня self hosted cyrus без интерфейса». Значит:

- **Канала «дашборд Skills page»/«Install curated» НЕ существует** для этого инстанса. Любые советы «поставь с дашборда / app.atcyrus.com / Manage scope» — НЕРЕЛЕВАНТНЫ. Не предлагать.
- **Cyrus-native скиллы** в этой сессии: `cyrus-skills:debug/implementation/investigate/summarize/verify-and-ship` — дефолтный набор рантайма. Кастомные Cyrus-скиллы для OSS-инстанса добавляются только правкой конфига cyrus-процесса на хосте (Railway) + редеплой/рестарт — не версионируется в проекте, общий на все репо инстанса.

**Два реальных канала кастомизации для этого сетапа:**
1. **Репо `.claude/`** — ЕДИНСТВЕННЫЙ версионируемый, едет с worktree, виден в PR, работает для любого Cyrus. Cyrus гоняет Claude Code внутри worktree → нативно подхватывает `CLAUDE.md`, `.claude/agents/`, `.claude/commands/`, `.claude/settings.json` (это явно задокументировано как «carries over»). `.claude/skills/*/SKILL.md` — нативная фича Claude Code, в Cyrus-доках в списке «carries over» НЕ перечислена → подхват под вопросом, проверять; гарантированно работают commands/agents/CLAUDE.md.
2. **Хост-конфиг cyrus на Railway** — для кросс-проектных вещей на ВСЕ репо инстанса; руками + редеплой, не версионируется. Pavel по сути гоняет один продукт → канал 1 (репо) почти всегда выигрывает.

**Категориальная ошибка тикета TRIP-23:** ECC / superpowers / engineering / agent-skills / design / ui-ux-pro-max / web-quality-skills — это **плагины Claude Code из маркетплейсов**, рассчитанные на **desktop Cowork / CLI** (namespacing `ecc:*`, `/ecc:configure-ecc`; см. [[triplanio-ecc-toolkit]]). **Cyrus НЕ умеет ставить сторонние плагин-маркетплейсы** ни из репо, ни из конфига. Нативный аналог в Cyrus = отдельные `SKILL.md` (имя + description-триггер + тело). Поэтому «положить ECC в репо для Cyrus» не сработает; ценность — переписать наш плейбук как свои SKILL.md/.claude-ассеты.

**MCP для self-hosted:** конфигурится файлом `.mcp.json` в корне репо (версионируется, секреты через `${ENV_VAR}`) — это канал, аналогичный репо-`.claude/`. У нас уже есть `.mcp.json` (TRIP-20 Sentry).

- **35 скиллов** в `.claude/skills/<source>-<skill>/SKILL.md` (неймспейс по источнику, чтобы не было коллизий `name:`): ECC×9 (affaan-m/ECC, MIT), superpowers×3 (obra, MIT), agent-skills×4 (addyosmani, MIT), web-quality×6 (addyosmani, MIT), design×5 (anthropics/knowledge-work-plugins, Apache-2.0), engineering×6 (там же, Apache-2.0), ui-ux-pro-max×1 (nextlevelbuilder, MIT, +data/scripts), example-frontend-design×1 (anthropics/skills, Apache-2.0).

- **★СМОУК-РЕЗУЛЬТАТ (TRIP-22, 2026-06-24): `.claude/skills/` НЕ автоподгружается self-hosted рантаймом.** В реальной сессии список доступных Skill-инструментов = ТОЛЬКО 5 `cyrus-skills:*` (debug/implementation/investigate/summarize/verify-and-ship). Ни один из 38 завендоренных `.claude/skills/*/SKILL.md` (ecc-*, superpowers-*, agent-skills-*, web-quality-*, design-*, engineering-*, ui-ux-pro-max, example-frontend-design) в Skill-список НЕ попал. Подтверждена догадка из доков.
- **А вот `.claude/agents/` и `.claude/commands/` ГРУЗЯТСЯ** (как и обещано «carries over»): 5 ECC-агентов (`ecc-planner`, `ecc-react-reviewer`, `ecc-typescript-reviewer`, `ecc-react-build-resolver`, `ecc-tdd-guide`) реально видны как subagent_type в Agent-туле; 6 команд `/ecc-*` лежат на гарантированном канале.
- **Вывод по TRIP-23:** чтобы ECC/superpowers/web-quality/design-плейбуки реально работали в Cyrus — их надо конвертить в `.claude/agents/` (как уже сделано для 5 ECC react/ts/tdd/plan) или `.claude/commands/`, а не оставлять как `.claude/skills/`. Вендоринг в skills = мёртвый груз для этого рантайма (полезен только если открыть файл руками через Read). Для FE/dev по факту автодоступны: дефолтные cyrus-skills (весь dev-цикл) + ECC react/ts-агенты и команды.
- **★РЕШЕНИЕ дизайн-скиллов (TRIP-22, 2026-06-24): через указатель в `CLAUDE.md` rule 15, а не конвертацию в агенты.** Pavel: «ты должен уметь обращаться к дизайн-скиллам, когда я прошу фронтенд/дизайн/редизайн». Поскольку файлы лежат на диске и читаемы, а `CLAUDE.md` автозагружается ВСЕГДА — добавлено rule 15 «Design / frontend skills (MANDATORY consult)»: на любой FE/дизайн/редизайн-задаче ДО написания UI-кода открывать через `Read` нужный `.claude/skills/<name>/SKILL.md` и применять. Список в rule 15: 5 design-* (critique/system/a11y/ux-copy/handoff) + example-frontend-design + agent-skills-frontend-ui-engineering + ui-ux-pro-max + web-quality-*. Дублировать в агенты НЕ стали (reuse-first rule 6: вендоренные файлы = единый источник, агенты-копии = дубль 1725 строк). Для React/TS/build по-прежнему ECC-агенты.
