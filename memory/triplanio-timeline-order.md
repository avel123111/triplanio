# Triplanio: порядок городов и варнинги «Нет переезда» в таймлайне

## Был баг (trip 49a1de05 на prod)
Варнинги «Нет переезда · A → B» связывали нехронологических соседей. Причина: ДВА независимых порядка — раскладка по дням (`visitForDay`, первый город, чей диапазон покрывает день) vs пары для варнингов (`sortVisits` по точному timestamp). На общих/перекрытых днях они расходились, одна кривая пара каскадом «съезжала» на все следующие (напр. Милан 06-10 12:00 вклинивался между Мадрид 06-10 00:00 и Канары → пары Мадрид→Милан, Милан→Канары).

## Фикс (2026-05-30, в общий неотпушенный релиз)
Единый источник порядка — один проход по упорядоченному списку городов.
- `src/lib/validation.js` `sortVisits`: тай-брейк при равном `start` → по `end` (раньше кончается = раньше в списке). Порядок: start → transit(by start,then end) → end.
- `src/pages/TripView.jsx` `TimelineLens`: рендер переписан с «один город на день» (`visitForDay`) на «CityHero по дню ПРИБЫТИЯ (start day)», допускается несколько Hero в один день. Перед каждым Hero — карточка трансфера (если есть prev→city) или варнинг «Нет переезда». `prevCity` ведётся сквозняком по всему проходу → «from» в варнинге всегда = город, нарисованный прямо выше. `visitForDay` больше не используется (dead).
- Модель (подтв. Pavel): город рисуется как Hero только в день прибытия; общий день улёта/прилёта (улетел 13/прилетел 13) — норма, `visitForDay`-приоритет НЕ меняли; 2 Hero в один день допустимы, между ними варнинг/трансфер.

## Остаток
- Настоящее ПЕРЕКРЫТИЕ дат (город вложен в диапазон соседа, как Милан внутри Мадрида в этом трипе) — это противоречивые данные; фикс делает картину согласованной (без каскада), но порядок таких городов остаётся по (start,end). Полное лечение — валидация дат при сохранении города (`end ≥ start`, без перекрытий с соседями) + правка данных трипа. Отдельная задача (не сделана).
- `order_index` (явный порядок городов) НЕ вводили — пока хватает ключа (start,end). Если понадобятся одинаковые start+end или ручной reorder — вводить явный индекс.
