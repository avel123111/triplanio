★РЕШЕНИЕ Pavel 2026-06-11 (Вариант B): редактор структуры трипа уходит от edit-mode/lock/draft к **live-записи в БД с серверным recompute**. Undo не нужен (явно сказано). Это разворот песочницы [[triplanio-sandbox-redesign]] / [[triplanio-edit-mode]].

**Почему:** референс Stippl (проверено в network через Claude in Chrome): live без лока/драфта — точечные GraphQL-мутации (`UpdateDestinationDuration{destinationId,newDuration}` дельта одного города; `addDestinationToTrip{...}` сразу реальный id) + **серверный recompute** + рефетч `Trip`. Нет websocket/realtime, нет version/etag → last-write-wins. Под этой моделью три бага редактора ([[triplanio-edit-mode]] dirty/двойная вставка/зависший лок) и проблема tmp-города исчезают сами.

**Ключевой разблок:** перенос `recompute` (формула `start=prevEnd+gap; end=start+nights`, `gap` из `transfers.day_change` через `dayChangeByTo`, сейчас ТОЛЬКО на клиенте в `TripStructureEdit.jsx`) на сервер + сделать каскад транзакционным side-effect'ом ЛЮБОЙ записи трансфера → **овернайт-трансфер (day_change) можно создавать из любого места** (таймлайн/панель/AI/бот), не только из edit-screen. Условие: правило «смежного» трансфера для gap (out-of-plan не вмешивается); см. [[triplanio-overnight-transfer-flag]] и [[triplanio-tmp-transfer-uuid-bug]].

**Артефакты:** ТЗ-файл `TRIP_LIVE_EDIT_SERVER_RECOMPUTE_TZ_2026-06-11.md` (outputs; «Triplanio docs» не примонтирована); Jira **Эпик TRIP-126** (проект TRIP, cloudId 73bde511-..., site triplanio.atlassian.net) держит полное ТЗ. Подзадачи: Ф1 TRIP-127 (Postgres recompute_trip + per-action RPC + триггер на transfers), Ф2 TRIP-128 (day_change anywhere), Ф3 TRIP-129 (выпил edit-mode + зачистка клиента), Ф4 TRIP-130 (сайд-эффекты/конкуренция), Ф5 TRIP-131 (выкат + зачистка сервер/БД), Ф6 TRIP-133 (ManualPlanner manual+AI на общую логику — убрать второй движок recomputeDates), Ф7 TRIP-134 (Live-UX: оптимистика/секвенс/ошибки/рефетч-без-мерцания). Бэклог (low, вне эпика): TRIP-132 (optimistic-concurrency guard). В КАЖДОЙ задаче — подробный чек-лист тестов (сценарий→ожидаемо) комментарием.

**Решения Pavel 2026-06-11:** (Q1) эпик+подзадачи; (Q2) concurrency-guard НЕ сейчас → отложено в TRIP-132, на старте last-write-wins; (Q3) точки ввода дат СТРОГО = trip `start_date` (`setTripStartDate`) + `nights` + `gap` (через day_change), `setCityDates` per-city НЕ делаем (даты производны); (Q4) recompute = **Postgres-функция (PL/pgSQL)** внутри per-action RPC, не JS-в-edge. Зачистка мёртвого кода расписана: клиент Ф3, сервер/БД Ф5 (save_trip_edit, лок-RPC+таблица trip_edit_lock), с обязательной проверкой реального неиспользования (recompute/buildDraft м.б. переисп. в ManualPlanner/таймлайне). **base44 НЕ релевантен этой фиче** (edit-mode/recompute/лок в base44-архитектуре не существовало) — base44-сверка из Ф0 убрана, несмотря на общий проект-rule [[feedback_base44_analysis_rule]] «анализ base44 перед кодом». Ф0 = деп-цепочка только по новой архитектуре.

**★ФИКС 0043 (2026-06-20, Вариант B) — овернайт start→first-transit теперь учитывается.** Был баг: овернайт-плечо (`day_change=true`) ИЗ start-города в первый transit/waypoint игнорировался → первый город стоял на дне ВЫЛЕТА, а не прилёта. Причина: цепочка анкорилась на `start_date` самого первого не-якоря И форсила его gap=0 (0027 [R1], зеркально клиент `layoutDates`). Решение (миграция **0043_recompute_anchor_first_leg.sql**): (1) новый хелпер `_trip_anchor_date(trip)` = дата вылета (UTC, `(start_datetime at time zone 'UTC')::date`) первого плеча из start-города → фолбэк first-non-anchor.start_date → trips.start_date → today; (2) `recompute_trip` берёт базу из него и БОЛЬШЕ НЕ форсит gap первого не-якоря (убран `if v_seen`) — gap считается для всех, включая start→first и овернайт между waypoints до first transit; (3) `reorder_cities`/`remove_city` пинят базу через `_trip_anchor_date`; (4) триггер `trg_recompute_on_transfer_upd` теперь фаерит и на смену `start_datetime`. Клиент-паритет 1:1: `tripDates.layoutDates` (убран forced-0) + `TripStructureEdit.buildDraft` (startDate = дата вылета start-плеча). **ДОП-ПРАВКА (2026-06-20):** шапка «Старт» (`TripStartControl`), дата старт-якоря «Отправление» (`GridEndpoint date=`) и `pickStart` читали рендер-`startDate = seq[0].start_date` (= старт ПЕРВОГО города = день прилёта) → показывали 15 вместо дня вылета 14; перенаправлены на `draft.startDate` (= дата вылета). ОТКРЫТО: header-диапазон `formatTripRange(draft.nodes)` всё ещё 15–22 (старт-якорь без даты) — день вылета 14 в диапазон не входит; чинится в рамках выпила/деривации trip start-date. **Применено prod+dev (functions/trigger).** Проверено на trip `950f33cb-…b2b473eba987` (Мадрид→Белград овернайт W64040): Белград 14→21 стал **15→22**, идемпотентно. **Эффект Варианта B:** старт +1 И конец +1 (ночи сохраняются), симметрично finish-якорю. **Последствие:** трипы с ночами, введёнными под багом, дадут варнинг рассинхрона с обратным рейсом (Белград end 22 vs рейс 21) — юзер уменьшает ночи. **ОТКРЫТО:** (a) бэкафилл остальных трипов НЕ запущен, ждёт решения Pavel (пока лениво при след. правке); (b) выпил trips.start_date/end_date — отдельным шагом (читатели: recompute_trip фолбэк, add_city); (c) дубль номера миграции 0040 в репо.
