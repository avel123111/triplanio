Trip 49a1de05-a05c-4a23-a853-407a32ac3f75 (prod) выявил архитектурную слабость рендера таймлайна в `src/pages/TripView.jsx` (новая архитектура). Корень — итинерарий, где два city_visit делят один календарный день (Милан 06-10→06-10 и Мадрид 06-10→06-13; Мадрид …→06-13 и Канары 06-13→06-17). Таймлайн местами полагается на date-range эвристики вместо FK-связей, и это рвётся при перекрытии.

ИСПРАВЛЕНО 2026-06-03 в `src/pages/TripView.jsx` (ветка dev, ждёт cherry-pick в main). Текущее поведение таймлайна:

**Привязка отеля к городу** — СТРОГО по FK, без fallback (решение Pavel 2026-06-03). `buildEventStream` кладёт `cityVisitId: h.city_visit_id` в hotel-события; `hotelsByVisit` матчит только `v.id === e.cityVisitId`. Нет совпадения → отель не цепляется к hero, город рисует стандартный варнинг «нет брони» (чек-ин/выезд при этом всё равно есть в стриме как строки). Прод-данные чистые: 14/14 отелей с валидным city_visit_id (0 null, 0 битых), регресса нет. Так публичный `ReadOnlyTimelineView` (groupBy h.city_visit_id) делал всегда. (Был баг: прежний `name OR date-range` в одном `.find()` брал ПЕРВЫЙ город дня — при перекрытии Милан/Мадрид отель Мадрида уезжал в hero Милана.)

**События вне окна трипа** — отдельные «плоские» дни. `outOfRangeDays` = даты событий вне [tripStart,tripEnd]; pre-trip дни рендерятся НАД start-anchor, post-trip — после end-anchor (helper `renderEventsDay`). Раньше дедлайн бесплатной отмены до старта трипа (06-05 при старте 06-08) молча терялся, т.к. `buildDayList` покрывает только [tripStart,tripEnd]. Продуктовое решение Pavel: показываем день события, затем start-city, затем дни трипа.

**Порядок внутри дня** — хронологический. На arrival-дне каждый блок [transfer/warning + CityHero] якорится временем входящего transfer (или start города); события дня с `_ms` меньше первого якоря рендерятся НАД блоком(ами), остальные — под. Раньше рендер был двухфазным (arrival-блок всегда первым), и «Выезд» 11:00 уезжал под перелёт 12:20.

Заметки: `ReadOnlyTimelineView` (public) — отдельная реализация и фактически ЭТАЛОН: всех трёх багов там нет. Привязка отеля по `groupBy(h.city_visit_id)`; единый плоский стрим, сортируемый по `sortMs` (город = событие с sortMs = время входящего transfer +1мс → хронология сама собой); `allDayKeys = tripRangeDays ∪ eventDayKeys` — дни событий вне окна трипа уже включаются (коммент прямо про дедлайн до старта). Мои правки привели TripView в соответствие с ним. Единственное расхождение: ReadOnly рисует cancel-событие при наличии только `free_cancellation_until` (без булева `free_cancellation`), TripView требует оба — и тут «правее» TODO TripView (нельзя показывать дедлайн бесплатной отмены для невозвратной брони). `visitForDay` в TripView — УДАЛЁН (был dead code). Все три бага проявляются ТОЛЬКО при перекрытии city_visit на один день.
