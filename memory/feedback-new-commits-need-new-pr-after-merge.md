Когда нужен мердж — ВСЕГДА должен быть **открытый** PR, в котором Pavel видит изменение (и Vercel-preview). Голый push в ветку — невидим.

**Грабли (TRIP-51, 2026-06-27):** PR #204 был смерджен Pavel'ом, после чего я запушил ещё один коммит (`b6c32b7`) в **ту же** ветку. Смердженный PR заново НЕ открывается → коммит повис без PR, Pavel «опять не видит PR», и Vercel-preview на него не строится. Пришлось завести отдельную ветку от свежего `origin/dev` + cherry-pick остатка + новый PR #206.

**Почему так:** репо мерджит **squash** (см. [[triplanio-dev-main-sync-no-squash]]) — после squash-мерджа коммиты исходной ветки по SHA в историю `dev` не попадают (там одна squash-копия). Поэтому продолжать пушить в уже-смердженную ветку нельзя: PR с неё показал бы уже смердженные правки повторно.

**Why:** для Pavel PR = единственная видимая в Linear/Vercel единица ревью; коммит без открытого PR = работа, которую он не может ни увидеть, ни задеплоить на preview. Это прямое усиление [[feedback-all-pushes-visible-linear-review.md]] и git-правила #5.

**How to apply:**
1. Один логический инкремент = один PR. Не досыпать коммиты в ветку после того, как её PR смерджен.
2. Если PR уже смерджен, а есть ещё правки → **новая ветка от свежего `origin/dev`** (`git fetch origin dev` → `git checkout -b <new> origin/dev`), перенести только нужный коммит (cherry-pick), push, **新ый** `gh pr create --base dev`.
3. После КАЖДОГО push — явный отчёт: номер+ссылка PR, его state (OPEN/MERGED), HEAD-SHA. Перед «готово» проверять `gh pr view <n> --json state` — если MERGED, а у меня есть свежий коммит, значит он висит без PR.
4. Не путать «запушил» с «открыл PR» — это разные вещи; цель достигнута только когда есть OPEN PR с изменением.
