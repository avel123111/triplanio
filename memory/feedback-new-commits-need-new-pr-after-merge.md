Когда нужен мердж — ВСЕГДА должен быть **открытый** PR, в котором Pavel видит изменение (и Vercel-preview). Голый push в ветку — невидим.

**Грабли (TRIP-51, 2026-06-27):** PR #204 был смерджен Pavel'ом, после чего я запушил ещё один коммит (`b6c32b7`) в **ту же** ветку. Смердженный PR заново НЕ открывается → коммит повис без PR, Pavel «опять не видит PR», и Vercel-preview на него не строится. Пришлось завести отдельную ветку от свежего `origin/dev` + cherry-pick остатка + новый PR #206.

**РЕЦИДИВ (TRIP-140, 2026-06-27):** ровно то же самое — PR #219 смерджен, я запушил коммит фиксов (`a45a12e`) в ту же ветку → Pavel «я не вижу pr опять». Исправил веткой `cyrus1/trip-140-fixes` от `origin/dev` + cherry-pick + новый PR #220. **Вывод: одной «реактивной» памяти мало — нужен ПРЕВЕНТИВНЫЙ гейт ПЕРЕД каждым push (см. ниже), а не починка постфактум.**

**РЕЦИДИВ №3 (TRIP-141, 2026-06-28):** PR #225 смерджен Pavel'ом, я запушил коммит визуальных правок по ревью (`4662903`) в ту же ветку, **не прогнав пре-push гейт** → Pavel «я не вижу опять блять твой pr». Исправил веткой `cyrus1/trip-141-cluster-visual-review-fixes` от `origin/dev` + cherry-pick `4662903` → новый PR #226. **Корень: гейт был записан, но я его не выполнил механически.** Триггер для рецидива — «быстрая правка по ревью сверху»: именно тут расслабляешься и дописываешь в старую ветку. ПРАВИЛО БЕЗ ИСКЛЮЧЕНИЙ: ЛЮБОЙ push (включая однострочную правку по ревью) начинается с `gh pr list --head <branch> --state all`. Особенно — когда между моим прошлым пушем и текущим Pavel успел дать фидбэк (он почти наверняка уже смёржил).

**Почему так:** репо мерджит **squash** (см. [[triplanio-dev-main-sync-no-squash]]) — после squash-мерджа коммиты исходной ветки по SHA в историю `dev` не попадают (там одна squash-копия). Поэтому продолжать пушить в уже-смердженную ветку нельзя: PR с неё показал бы уже смердженные правки повторно.

**Why:** для Pavel PR = единственная видимая в Linear/Vercel единица ревью; коммит без открытого PR = работа, которую он не может ни увидеть, ни задеплоить на preview. Это прямое усиление [[feedback-all-pushes-visible-linear-review.md]] и git-правила #5.

**How to apply:**

**★ ПРЕ-PUSH ГЕЙТ (обязателен ПЕРЕД каждым `git push`, не после):** прежде чем пушить в текущую ветку, проверь, не смёржен ли уже её PR:
```
gh pr list --head "$(git branch --show-current)" --state all --json number,state
```
- Если есть PR со `state: MERGED` (или ветка уже вливалась) → **СТОП, не пушить в неё.** Сразу: `git fetch origin dev` → `git checkout -b cyrus1/<task>-<short> origin/dev` → `git cherry-pick <новые коммиты>` → `git push -u origin <new>` → `gh pr create --base dev`.
- Если PR `OPEN` → пуш в неё обновляет тот же PR (ок), но всё равно отпишись в тред (см. п.3).
- Если PR нет вовсе → push + `gh pr create`.

**Эвристика без gh:** как только Pavel написал «смёржил / влил / merged» ИЛИ ты видел мердж — считай ТЕКУЩУЮ ветку «мёртвой» для новых коммитов. Любая следующая правка по этой задаче = новая ветка от свежего `origin/dev`. Не дописывай в смердженную ветку даже один коммит.

1. Один логический инкремент = один PR. Не досыпать коммиты в ветку после того, как её PR смерджен.
2. Перенос правок на новую ветку — только cherry-pick нужных коммитов (не merge), база = свежий `origin/dev`.
3. После КАЖДОГО push — явный отчёт в тред: номер+ссылка PR, его state (OPEN/MERGED), HEAD-SHA, что изменилось. Перед «готово» ещё раз `gh pr view <n> --json state` — если MERGED, а у меня есть свежий локальный коммит сверху, значит он висит без PR → чини по гейту выше.
4. Не путать «запушил» с «открыл PR» — цель достигнута только когда есть **OPEN** PR с изменением, видимый в Linear.
