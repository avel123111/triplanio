**Цель (2026-06-03).** Привести все варнинги/ошибки к единообразию и заблокировать любое сохранение, нарушающее логику таймлайна/структуры трипа. Сейчас валидация размазана по 4 несвязанным механизмам (ad-hoc canSave в EventEditDialog, field-level red `<p>`, soft-плашки hotelWarnings/activityWarnings/transferWarnings, и отдельно computeTripValidation в Edit Mode) — разный вид и разный вердикт по одним данным; много дыр (напр. отель/активность сохраняются с пустыми датами, т.к. canSave их не требует).

**Решения Pavel (зафиксированы):**
1. **Out-of-bounds = жёсткая ошибка (error), блок сохранения.** Отель вне дат города, активность вне дат, переезд не стыкуется по дням — нельзя сохранить (без override).
2. **Все start И end обязательны, предзаполнены, нельзя очистить — только менять.** Пустую дату/время сохранить нельзя (error) + поля всегда сидят из контекста (visit/trip).
3. **Enforce пока только на фронте** (Ф0–Ф3). DB-констрейнты/save-RPC (Ф4) — отложено.
4. **Показ: инлайн под полями + единая панель.** Field-ошибки под полями (единый стиль), кросс-полевые/структурные — в одной панели над футером. Toast только для сетевых сбоев, не для валидации.

**Целевая архитектура (Вариант A).** Единый чистый модуль `validateEntity(kind, draft, ctx)` в lib/validation.js → `{level,code,field?,message}[]`; его едят и модалка (canSave = нет error), и Edit Mode, и таймлайн. Свернуть туда hotelWarnings/activityWarnings/transferWarnings/tripWarnings (дубли выпилить). Общий `<IssuesPanel>` + хук useEntityValidation.

**Архитектура движка:** `validateEntity(kind,draft,ctx)` (модалка, L1+L2) + `validateTrip(draft)` (Edit Mode, L3 — рефактор computeTripValidation поверх validateEntity) + `primaryIssues()` (таймлайн, ≤1 issue/сущность). Issue несёт `entityId/fromId/toId` (для Edit Mode подсветки) + `field` (для модалки). Edit Mode R3: гейт `blocked = issues.some(error)` (warnings не блокируют) — ждёт подтверждения.

**AI-парсинг (решение Pavel):** НЕ делать отдельных проверок на AI-ввод. Распарсенные данные — обычный черновик: мапим как есть, прогоняем через тот же validateEntity, поля НЕ скипаем (даже невалидные) — заполняем и показываем стандартные варнинги. Убрать из handleHotelExtract/handleTransferExtract: setIf-скип, typeof-гарды, aiEndpointWarn (его роль берут стандартные OOB/TR_DEP_DAY/TR_ARR_DAY). Нюансы: закрытые наборы (transport/currency) → из набора или дефолт; невалидная AI-дата → остаётся предзаполненный дефолт; город ре-резолвим через гео (координаты не от AI). Один гейт защищает и ручной ввод, и AI.

**i18n (Ф0.5, ДО локализации валидации):** неймспейс code-aligned — `validation.<CODE>` + `confirm.<CODE>.title/body`, ключ 1:1 с кодом. Usage-аудит локалей: 110 валидац./confirm-ключей, ~52 ОРФАНА (0 использований, в т.ч. в redesign) → выпил. Дубли неймспейсов (member/members, settings.plan_portal vs sub, 3× trip.delete*) свести к каноническим. Текущую локализацию валидац./confirm-ключей НЕ делать вслепую — переименуются/выпилятся.

**Показ ошибок (решение Pavel): ГИБРИД.** Инлайн-ошибка/рамка под полем — после касания этого поля (touched); панель-сводка снизу — только после нажатия Save. Свежая форма чистая. Кнопка Save кликабельна при ошибках (не disabled) → по клику показывает панель + фокус на первую ошибку, не сохраняет. Реализовано в EventEditDialog (Ф2): touched-Set + submitted-флаг, displayIssues gating, ValidationUI (FieldError/IssuesPanel/fieldHasError), .tv-invalid рамка в app.css.
