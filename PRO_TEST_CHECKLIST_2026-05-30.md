# Чек-лист тестирования: платежи и Pro-функционал (полный, перед продом)
_2026-05-30. Глобальная проверка всего платного функционала, а не только последних правок._

## 0. Преднастройка (без этого часть тестов не пройдёт)
- [ ] Stripe: у продуктов **pro_trip** и **pro_yearly** задан `default_price`; тестовая цена €0.05 **архивирована** (иначе checkout pro_trip = 500).
- [ ] Stripe webhook-эндпоинты (LIVE и TEST) подписаны на события: `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, **`charge.refunded`**, **`charge.dispute.created`**.
- [ ] Supabase secrets в каждом проекте: `STRIPE_SECRET_KEY` (prod=sk_live, dev=sk_test), `STRIPE_WEBHOOK_SECRET` (свой), `PUBLIC_APP_URL` (prod=`https://www.triplanio.com`).
- [ ] Тест-аккаунты: владелец, админ, зритель; тест-карта `4242 4242 4242 4242`.
- [ ] Подготовить 3 трипа: **FREE** (владелец без подписки, не pro_trip), **PRO-trip** (`is_pro_trip=true`), **PRO-через-подписку** (владелец с активной подпиской).
> Примечание: новый трип теперь стартует со всеми pro-линзами ВЫКЛ; существующие трипы тоже скрывают календарь/бюджет/чат пока аддон не включён — это ожидаемо, не баг.

## A. Базовая матрица доступа
- [ ] Free-аккаунт: 1 активный трип; базовые линзы (таймлайн/карта/документы) доступны; pro-фичи закрыты.
- [ ] Pro-подписка (monthly/yearly): безлимит трипов; pro-фичи во ВСЕХ своих трипах.
- [ ] Pro-trip: pro-фичи в ЭТОМ трипе для всех участников; на лимит трипов не влияет.

## B. Оплата — подписка (monthly / yearly)
- [ ] `/pro` без tripId: показаны monthly + yearly (pro_trip скрыт); цены тянутся из Stripe.
- [ ] Checkout monthly тест-картой → возврат в приложение.
- [ ] Webhook: `users.subscription_status='pro'`, `subscription_end_date`=+30 дн; строка `trip_subscriptions` (pro_monthly, active); запись в `stripe_events`; уведомление «Pro activated».
- [ ] Yearly: end_date +365; профиль показывает «Pro Yearly».
- [ ] Безлимит: Pro-юзер создаёт 2+ активных трипа.
- [ ] Дубль-подписка: повторный checkout при активной → 409 (или редирект в биллинг-портал), вторая не создаётся.
- [ ] Race-гард: быстрый повторный клик в течение 15 мин → «платёж обрабатывается».
- [ ] Биллинг-портал из профиля открывается (управление картой/планом).
- [ ] Отмена в портале → `subscription.updated(cancel_at_period_end)`: доступ до конца периода, профиль «Pro отменена — действует до …»; после конца периода → free.

## C. Оплата — pro_trip (разовый)
- [ ] FREE-трип, **владелец**: в сайдбаре «Апгрейд трипа» → `/pro?tripId=…` → виден план pro_trip.
- [ ] Checkout pro_trip → `trips.is_pro_trip=true`; строка trip_subscriptions (pro_trip, `stripe_payment_intent_id` заполнен); трип сразу Pro для всех участников.
- [ ] Сумма `amount_paid` = реальной цене (после фикса default_price).
- [ ] **Не-владелец** с прямой ссылкой `/pro?tripId=<чужой>`: план pro_trip НЕ показывается (подписку купить можно, pro_trip — нет).

## D. Возвраты / чарджбэки (после добавления событий в Stripe)
- [ ] Рефанд pro_trip-платежа → `charge.refunded` → `trips.is_pro_trip=false`, строка `status='refunded'`; трип снова free.
- [ ] Рефанд подписочного платежа → `users.subscription_status='free'`.
- [ ] Спор `charge.dispute.created` → доступ снят, `status='disputed'`.
- [ ] Идемпотентность: повтор события не ломает (запись в `stripe_events`).

## E. Модалки результата оплаты (единый показ в Layout)
- [ ] Успех (возврат `?stripe_status=success`): ОДНА модалка «Подписка активирована», чип «Pro Monthly · €X/мес» если цена известна (иначе только название), кнопка «Вернуться в Triplanio». Двух модалок одновременно НЕТ.
- [ ] Отмена/ошибка (`?stripe_status=cancel`): модалка «Оплата не прошла» (с кодом, если есть), «Закрыть» / «Повторить оплату» → ведёт на `/pro`.

## F. Лимит трипов (free = 1 активный)
- [ ] In-app «Новый трип» при лимите → **модалка Variant D** (hero «Планируй сколько угодно», колонки Free/Pro, «Не сейчас» / «Посмотреть тарифы» → `/pro`).
- [ ] Прямая ссылка на **ручной** планировщик при лимите → полноэкранный блокер.
- [ ] Прямая ссылка на **ИИ**-планировщик при лимите → **тот же полноэкранный блокер** (не модалка).
- [ ] **Серверный бэкстоп:** прямой `rpc('create_trip')` free-юзером с активным трипом → ошибка `TRIP_LIMIT_REACHED`, трип НЕ создаётся.
- [ ] Прошедший трип освобождает слот.

## G. Гейтинг Pro-фич по ролям × состоянию трипа (ключевое)
Проверить каждую ячейку (owner / admin / viewer):
- [ ] **FREE-трип, владелец:** сайдбар «Апгрейд трипа»; ИИ-парсер в event-dialog = locked → «Перейти к Pro» ведёт в `/pro`.
- [ ] **FREE-трип, участник (admin/viewer):** вместо «Апгрейд трипа» → «Подключает владелец» → **модалка «обратитесь к владельцу»** (без кассы). ИИ-парсер locked → клик → та же инфо-модалка. ← главный сценарий «не ведём к кассе впустую».
- [ ] **PRO-trip / PRO-через-подписку владельца:** все участники видят бюджет/чат/календарь (если аддоны вкл), ИИ-парсер доступен, баннера апгрейда нет.
- [ ] **Участник со СВОЕЙ подпиской на FREE-трипе:** pro-фичи НЕ открыты (определяется владельцем/трипом, не смотрящим).

- [ ] Settings → «Опциональные фичи»: Календарь (не Pro), Бюджет (Pro), Чат (Pro), Telegram (Pro), Совместный выбор отелей («Скоро», выкл). Пунктов «ИИ-помощник» и «Документы» НЕТ.

## L. Таймлайн: порядок городов и варнинги «Нет переезда» (свежий фикс)
- [ ] CityHero показывается в день **прибытия** города; «from» в варнинге трансфера = город, нарисованный непосредственно выше (всегда совпадает визуально).
- [ ] Нет «съезжающих»/нехронологичных пар (типа «Милан → Канары», когда выше Мадрид).
- [ ] Кейс граничного дня (улёт 13 / прилёт 13): порядок городов корректный, варнинг между правильными соседями.
- [ ] Кейс одного дня (город1 9–12, город2 12–12, город3 12–15): на 12-е — два Hero (город2, затем город3) с варнингом/трансфером между; город1 на 12-е только как «from».
- [ ] Существующий трансфер между городами → показывается карточка трансфера вместо варнинга.
- [ ] Варнинг перед финишем (последний город → Финиш), если нет трансфера.
> Замечание: трипы с РЕАЛЬНЫМ перекрытием дат городов (один вложен в диапазон соседа) показываются согласованно, но это противоречивые данные — лечится правкой дат (валидация дат — отдельная задача, в этот релиз не входит).
