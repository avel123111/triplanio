## 0. Решения Pavel (2026-06-04) — фиксируем как базу

1. **Брони пишутся в БД сразу** (как сейчас). Панели — это замена модалок по UI, а не перенос броней в песочницу. В драфте-песочнице по-прежнему стейджится **только структура** (порядок городов, даты, ночи, добавление/удаление), сохраняется через `save_trip_edit`. Отель/активность/переезд при создании/правке пишутся в БД немедленно + refetch.
2. **Охват — только редактор структуры** (`TripStructureEdit.jsx`). `EventModal`/`EventEditDialog` остаются для просмотра/правки в `TripView`, `PublicTrip`, `BudgetLens`, `ScreenMap`. Панели и модалки временно сосуществуют → **обязательное условие: общая логика, разная оболочка** (см. §4).
3. **Переезды — модель как сейчас (FK `from_city_visit_id`/`to_city_visit_id`), мы их сами не удаляем.** Соседство определяется по `position` (`orderIndex`: переезд «в плане», когда `toIndex === fromIndex + 1`). Если города переезда **не соседи** (или повисли) → переезд **не прячем и не блокируем сохранение**, а показываем в отдельной зоне **«Переезды вне плана»** (предложение по месту — §6). Снапшот-модель `from_city`/`to_city` из sandbox-дока — **отложена** (будет отдельным релизом, перепишет эту часть).
4. **Варнинги — пока только инлайн** (бейджи на строках + `WarnNote` внутри панелей). Глобальный `ConflictsPanel` **ничего не удаляем** — оставляем компонент и его место под картой, просто ведущая поверхность — инлайн. (Решение «инлайн-only» можно докрутить позже без потери кода.)

## 3. Целевой UX редактора (склейка дизайна + реальной архитектуры)

Левая колонка — **конечный автомат панелей** (как `panel.view` в мокапе):

```
list → city(segId) → { hotelView(id) | activityView(id) | transport(legKey) }
list → cityadd
list → pick({kind,ctx}) → form({kind,mode,ctx,data})
*View → form(edit)   (стек возврата _back)
```

- **list** — таблица: эндпоинты, города (с инлайн отелём+активностями), коннекторы-переезды между соседями, зона «вне плана», «+ добавить точку», трей удалённых. Header трипа + кнопки undo/reset/cancel/save + индикатор варнингов сверху (как сейчас).
- **city** — `CityPanel`: фото, даты/ночи (правят структуру драфта!), прибытие/отправление (переезды), отель, активности.
- ***View** — read-панели; «Изменить» → `form(edit)`, «Удалить» → confirm → delete в БД.
- **pick/form** — создание: «Развилка» → форма (AI + поля + пересадки) → save в БД → возврат на city/list + invalidate content.

Карта справа остаётся; `focus` следует за открытой панелью (есть в мокапе — переносим). `ConflictsPanel` под картой остаётся (Q4).

## 4. Архитектурная стратегия — ОДНА логика, ДВЕ оболочки

Главный риск редизайна — расхождение «панель vs модалка», т.к. модалки остаются (Q2). Решение: **извлечь тела из модалок в presentational-компоненты, не привязанные к Dialog-оболочке.**

### 4.1 Рефактор `EventModal` (view)
- Вынести `HotelBody`/`TransferBody`/`ActivityBody`/`ServiceBody` + meta-strip + docs в `EventViewBody` (чистый рендер по `kind/entity/visit/...`).
- `EventModal` = Dialog-обёртка над `EventViewBody` (текущие call-sites не меняются).
- Панель `*View` = панельная обёртка (`PanelShell`) над тем же `EventViewBody`.

### 4.2 Рефактор `EventEditDialog` (create/edit)
- Вынести тело формы (`HotelFields`/`TransferFields`/`ActivityFields` + `SegmentsEditor`/`LayoverToggle` + AI-парс) и **payload-билдеры** + `saveLayoverChain` в `EventEditBody` (контролируемый стейтом снаружи или с колбэком `onSaved`).
- `EventEditDialog` = Dialog-обёртка (текущие call-sites не меняются).
- `FormPanel` = панельная обёртка над `EventEditBody`. Save → те же билдеры → БД (немедленно, Q1) → `onSaved` → `invalidate(TRIP_CONTENT_KEY)` → `nav.city/list`.

### 4.3 Контроллер загрузки
- Логику `SourceViewLoader` (load row by id + fromVisit/toVisit) обобщить так, чтобы её мог дёргать и модальный путь, и панельный (хук `useEntitySource(kind,id)`).

> Принцип: **ни одной копии полей/билдеров/парсинга.** Панели = новые оболочки вокруг существующих тел. Это снимает риск «починили в модалке — забыли в панели».

## 6. Переезды и зона «вне плана» (Q3)

- Коннектор между **соседними** (`orderIndex` n→n+1) рисуется как сейчас (`GridTransfer`): есть переезд → плашка с типом/датой (+ мягкий бейдж если даты не сходятся); нет → ghost «+ переезд». Дата-несостыковка = **warn, не блок**.
- **`TR_NOT_ADJACENT` понижаем error→warn** в `validation.js` (иначе несоседний переезд блокирует save, что противоречит «не блокируем»). Сверить с `VALIDATION_RULES_2026-06-03.md` и таблицей `primaryIssues`.
- **Зона «Переезды вне плана»** — предлагаемое место: отдельный блок внизу левого списка, **рядом/над `RemovedTray`** (тот же паттерн «трей»). Каждый элемент: `from → to`, дата, мягкий бейдж «вне маршрута», клик → `TransferView`. Туда попадают переезды, чьи `from/to` города существуют, но **не соседи по position** (или один из городов удалён → «висячий»). Мы их **не трогаем автоматически** — пользователь сам решит (перетащить города, удалить переезд, или оставить).
  - *Альтернатива (на потом):* инлайн между фактическими городами, даже если они не соседи, с «обводкой вне плана» — но это сложнее по верстке грида; начинаем с трея, дальше разберёмся (по слову Pavel).
