# Triplanio — единый свод правил валидации (Ф0)

Статус: **предложение к согласованию.** Это контракт для `validateEntity()` и единого показа. Решения Pavel зашиты: out-of-bounds = **error**; все `start`/`end` обязательны, предзаполнены, неудаляемы; enforce пока только на фронте; показ = инлайн под полями + единая панель.

---

## 1. Модель issue (единая для всего)

```ts
type Issue = {
  level: 'error' | 'warning';   // error блокирует сохранение, warning — нет
  code: string;                 // стабильный код, напр. 'HOTEL_CHECKIN_OOB'
  scope: 'field' | 'entity' | 'structure';
  field?: string;               // для инлайн-показа и подсветки (напр. 'checkInLocal')
  message: string;              // всегда через t(), без хардкода
  ctx?: Record<string, string>; // {city}, {from}, {to} для интерполяции
};
```

`validateEntity(kind, draft, ctx)` → `Issue[]`. Один движок зовут: модалка, Edit Mode, таймлайн.
`canSave = issues.every(i => i.level !== 'error')`.

---

## 2. Где что проверяем (уровни)

| Уровень | Что проверяет | Где выполняется | Блокирует |
|---|---|---|---|
| **L1 · поле** | required, время задано, порядок внутри сущности (end>start) | модалка (синхронно на черновике) | да (error) |
| **L2 · сущность-в-контексте** | границы родительского города, стыковка дней с городами | модалка + Edit Mode | да (error) |
| **L3 · структура трипа** | соседство переездов, дубликаты, «висящие» без города, наложение/разрыв городов | Edit Mode (полный прогон) + модалка (подмножество про эту сущность) | error — да, warning — нет |
| **Бэк (Ф4, отложено)** | те же инварианты как CHECK/RPC | Supabase | да |

---

## 3. Как и где показываем (единый контракт)

| scope | Где на экране | Вид |
|---|---|---|
| `field` | **под полем** (`<FieldError>`) + красная рамка поля | красный текст, единый стиль везде |
| `entity` / `structure` | **единая панель `<IssuesPanel>` над футером** | error — красная строка, warning — янтарная; клик → фокус/подсветка поля по `issue.field` |

Правила показа:
- Кнопка «Сохранить/Создать» **disabled ⟺ есть хоть один `error`**.
- **Toast** — только для сетевых сбоев сохранения (`onError` мутации), НЕ для валидации.
- Поля `start/end` всегда предзаполнены из контекста; пустое значение = `error` (поле красное) + ресид к дефолту → «удалить нельзя, только поменять».
- Один стиль во всех модалках (event/city/budget) — никаких жёлтых EN-плашек и разрозненных текстов.

## 4. Правила по сущностям

Легенда уровня: **E** = error (блок), **W** = warning (показ, не блок). Сообщения — предварительные (пойдут через `t()`).

### 4.1 Hotel (контекст: city visit)
| code | Ур. | scope | Условие | field | Сообщение |
|---|---|---|---|---|---|
| HOTEL_NAME_REQUIRED | E | field | название пустое | name | Укажи название отеля |
| HOTEL_CHECKIN_REQUIRED | E | field | заезд пуст / без времени | checkInLocal | Укажи дату и время заезда |
| HOTEL_CHECKOUT_REQUIRED | E | field | выезд пуст / без времени | checkOutLocal | Укажи дату и время выезда |
| HOTEL_ORDER | E | field | выезд ≤ заезд | checkOutLocal | Выезд должен быть позже заезда |
| HOTEL_CHECKIN_OOB | E | entity | день заезда < дня прибытия в город | checkInLocal | Заезд раньше прибытия в {city} |
| HOTEL_CHECKOUT_OOB | E | entity | день выезда > дня выезда из города | checkOutLocal | Выезд позже выезда из {city} |
| HOTEL_NO_CITY | E | structure | нет city_visit_id | — | Отель не привязан к городу |
| HOTEL_OVERLAP | W | structure | пересечение с другим отелем того же визита | — | Пересекается с отелем «{name}» |

> Меняем относительно текущего: check-in/out становятся **обязательны** (раньше canSave требовал только name); OOB B1/B2 повышаем soft→**E**.

## 5. Открытые развилки (нужно твоё «да/нет»)

1. **CITY_OVERLAP** — предлагаю **error** (наложение городов ломает линейный таймлайн). Ок?
2. **CITY_GAP** (разрыв > 1 дня) — предлагаю оставить **warning** (незапланированные дни — не структурная ошибка). Ок?
3. **DUP_TRANSFER** — **warning** или хочешь жёсткий блок (**error**)?
4. **Service внутри дат трипа** — добавлять мягкую проверку (**W**) или не трогать?

Всё, что в таблицах помечено **E**, после внедрения нельзя будет сохранить — это и закрывает дыры (в т.ч. твой пример с очисткой предзаполненных дат отеля/активности).

## 6. Что меняется относительно «сейчас» (сводка перехода)

- Soft-плашки `hotelWarnings/activityWarnings/transferWarnings` (B1/B2/C1/C2/D2) → становятся **error** в общем движке, показываются инлайн/в панели единым стилем.
- `canSave` перестаёт быть ad-hoc — это просто «нет error».
- Все `*_REQUIRED` на даты/время — новые жёсткие правила (закрывают дыры пустых дат).
- `tripWarnings` и осиротевшие хелперы — удаляются (их роль берёт `validateEntity` + `computeTripValidation`, сведённые к одному набору правил).
- Тексты — все через `t()` (RU/EN/ES), EN-плашки уходят.
